# Review: 通用导入中心与页面内目标支派选择

## Correctness

- “导入管理”保留为一级菜单和页面名称。
- `ImportPage` 已成为通用导入中心，而不是人物导入组件的薄封装。
- 人物导入标记为已支持；关系、字辈和来源资料导入明确标记为规划中并禁用。
- 当前页面根据所选宗族加载支派列表，并提供可搜索的目标支派选择器。
- `PersonImportWorkspace` 不再直接读取 `WorkspaceContext`，预览和创建批次均使用页面传入的宗族与支派。
- 切换支派会清空文件、预览和重复确认，防止使用旧预览跨支派提交。
- 导入任务面板上移到通用中心，创建批次后会刷新任务列表。
- 页面选择会同步工作区支派，使现有任务面板按同一支派过滤，但用户不再需要先到其他页面设置支派。

## Readability

- 导入类型元数据集中在 `import-type-registry.ts`，避免类型文案和可用状态散落在页面中。
- `ImportPage` 负责页面级类型、支派作用域和任务刷新。
- `PersonImportWorkspace` 只负责人物模板、文件、映射、预览和创建批次。
- 页面文案统一使用“目标支派、人物导入、规划中”等业务表达，不展示技术 ID。

## Architecture

- 页面级状态保留在 `ImportPage`，符合局部状态优先原则。
- 跨页面 Workspace 仅承担可选默认值和既有任务面板兼容，不再作为人物导入组件的隐式输入。
- 通用任务面板与类型专用导入表单解耦，为后续关系导入复用任务生命周期预留空间。
- 本次未修改后端、数据库、审核流程或 API 契约，控制在前端垂直切片内。

## Security

- 页面只展示支派名称，支派 ID 仅作为接口参数在内部使用。
- 后端原有支派权限校验保持不变，前端选择不能绕过后端鉴权。
- 文件中的支派字段仍不会覆盖页面明确选择的目标支派。
- 宗族变化时清空旧支派，避免跨宗族误用旧作用域。

## Performance

- 进入导入中心时只请求一次当前宗族支派列表。
- 支派名称通过内存数组查找，不引入逐项请求或 N+1。
- 规划中导入类型来自静态注册表，不增加网络请求。
- 任务列表继续使用现有服务端分页和支派筛选。

## Verification

临时定向 GitHub Actions 已全部通过，验证后已删除工作流：

- `npm run api:check`
- 导入中心相关文件定向 TypeScript 检查
- `npm run build`
- 默认 Commercial Frontend Build

默认 Backend CI 中 Java 全量测试和 PostgreSQL 启动检查仍失败；本 PR 未修改任何后端文件，前端构建已通过，失败属于仓库现有后端检查问题。

## Remaining Work

- 后续关系导入应通过类型注册表接入，并复用通用任务面板。
- `ImportJobManagementPanel` 内部的失败行修正表单仍是人物专用，关系导入落地前需要抽取按类型渲染的修正器。
- `importType` 与 `fileFormat` 的后端契约拆分应作为独立 API 演进任务处理。
