## 设计方案

1. 页面保持单一一级导航，但内部拆为总览、筛选列表、详情 Drawer 和编辑 Modal。
2. 文化 API 服务统一封装路径、查询参数和生成 DTO，页面组件不直接散落拼接接口。
3. URL 使用 `cultureKeyword/category/branch/status/privacy/hasSource/featured/sort/page/item` 等模块前缀参数，避免污染其他页面状态。
4. 每次列表和详情请求分配递增 request token；宗族、筛选或选中对象变化后，旧响应不得回写新状态。
5. 按每条记录的 `allowedActions` 渲染动作；按钮隐藏只用于减少误操作，所有失败仍显示后端真实错误。
6. 正式资料编辑继续调用 PUT，由 #168 后端自动创建变更审核；草稿/驳回记录保存后直接刷新详情。
7. 来源与附件只展示后端返回的可见字段；不从来源列表、日志或 Diff 反推 sealed 内容。

## 已知风险与处理

- **接口形态差异**：文化分页使用 `items + page`，不能套用通用 `records`；服务层严格按生成类型解析。
- **旧响应覆盖**：API Client 的 GET 去重不等于取消，页面必须额外校验请求 token 和当前 clan/filter key。
- **正式数据误改**：页面明确区分“保存草稿”和“提交正式变更申请”，以后端返回消息为准。
- **隐私旁路**：无权或 sealed 的 403/404 不展示对象标题、来源或附件占位细节。
- **页面体积**：按状态、服务、列表、详情和表单拆分，避免再次形成大型单文件。
- **高风险误触**：归档、删除和精选变更使用 Modal/Popconfirm，展示影响对象并要求必要原因。
