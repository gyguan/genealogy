## DEFINE

### 目标

将数据导入页面重构为“导入任务查询 + 查询结果”双 Card 结构，并将新建导入收敛为弹窗选择对象、下载模板和上传预检入口。

### 范围

- 重组 `ImportPage` 页面骨架与查询状态。
- 复用现有导入工作区、异步执行、历史、失败修正和审核能力。
- 新增或调整页面级组件与样式。
- 更新数据导入相关聚焦测试和 Playwright E2E。

### 非目标

- 不修改 API、后端、数据库、权限和审核语义。
- 不修改导入模板字段、预检规则和批次创建接口。
- 不引入新依赖或第二套 UI 基础组件。

## 实现结果

1. 主页面只保留“导入任务查询”和“查询结果”两个主 Card。
2. 查询条件统一为导入对象、导入状态、文件名/任务编号、任务创建时间，支持 URL 恢复。
3. 导入对象和状态下拉支持多选、全选、清空和搜索。
4. 任务列表合并展示对象、文件、任务编号、目标范围、阶段、处理结果与上下文操作。
5. 新建导入弹窗支持人物、关系、来源选择，下载 XLSX/CSV 模板，并进入上传预检 Drawer。
6. 现有预检和批次创建工作区保持复用；批次创建后关闭 Drawer 并刷新任务列表。
7. 任务详情、暂停、继续、取消、重试保持可用；已完成任务可进入导入记录 Drawer。
8. 导入记录、失败修正与审核提交、审核历史迁入记录 Drawer，不再在主页面平铺多层 Card。
9. 390px 视口切换为任务记录卡片，核心查询、新建与操作不依赖横向滚动。

## 风险与控制

1. 现有页面同时管理创建、执行和历史状态；重组时复用现有 API、预检工作区和治理组件，没有复制领域规则。
2. 当前任务列表接口未提供原型全部组合筛选参数；页面从现有接口加载最近 200 条并在前端执行组合过滤，服务端总数超过当前加载量时明确提示用户缩小范围。未伪造新 API 契约。
3. 正式数据仍通过导入草稿、校验、审核后生效，未改变 P0 导入红线。
4. 本地环境无法解析 github.com；通过 GitHub Connector 修改，使用 GitHub Actions 完成实际依赖环境验证。
