## 1. Correctness

### 已确认

- CSV/XLSX 共用唯一模板定义，表头数量、名称和顺序严格校验。
- 仅有表头、没有数据行的文件会被拒绝，不创建空批次。
- 人物按 `clanId + personCode + deletedAt is null` 唯一匹配；0 条或多条均进入可修正失败行。
- 父子、母子、配偶映射到现有关系模型，父母性别规则明确。
- 自关联、数据库重复、代次冲突和祖先循环复用 `RelationshipApplicationService`。
- 文件内重复和文件内血缘循环在预览阶段额外校验。
- 失败行修正使用乐观锁，成功后生成关系草稿并重算批次状态。
- 审核通过统一转正式，驳回恢复草稿；配偶正反向关系同步状态。
- 人物导入同步写入通用草稿目标，历史人物导入和修正链路测试通过。

### Review 中已修复

- 修复仅有模板表头时生成空批次的问题。
- 补充关系失败行修正与版本冲突测试。

## 2. Readability

### 已确认

- 关系模板、文件策略、导入处理、失败行修正分别由独立服务承载。
- DTO、错误码和前端字段使用统一的 `fromPersonCode/toPersonCode/relationshipType/description` 语义。
- 通用任务中心只负责按 `importType` 选择修正表单，没有复制第二套任务页面。
- 执行看板和 OpenAPI 明确说明批次管理支派与关系双方支派权限的区别。

### 非阻断建议

- 当第三种导入类型落地时，可进一步将审核提交和审核生效的 `importType` 分派抽取为 Handler 注册表，避免分支继续增长。

## 3. Architecture

### 已确认

- 复用 `import_job`、`import_job_row`、错误行、审核任务和操作日志模型，没有新建平行批次体系。
- `import_job_row` 通过 `draft_target_type/draft_target_id` 支持多业务草稿，保留 `draft_person_id` 兼容窗口。
- 关系业务规则仍由 `relationship` 领域服务负责，导入模块不复制关系合法性算法。
- `RelationshipImportTypeDefinition` 已注册模板、解析、校验、修正、草稿和审核生效扩展点。
- Flyway 采用新增时间戳版本，历史迁移未修改，治理检查通过。

### 非阻断建议

- 兼容窗口结束后，可在独立 Issue 中移除 `draft_person_id` 和旧构造器兼容逻辑。

## 4. Security / Privacy

### 已确认

- 文件只接受人物业务编码，不暴露或接受内部人物 ID。
- 批次创建先校验管理支派写权限。
- 解析人物后、姓名进入预览或失败行数据前，先校验关系双方支派权限。
- 无权限预览只返回用户自己提交的业务编码和错误，不返回解析到的人物姓名。
- 正式创建继续复用关系领域的双方支派权限、同宗族和关系合法性校验。
- 提交审核再次校验批次管理支派和关系双方支派权限。
- 提交人不能审核自己的批次，沿用统一审核中心规则。
- 操作日志不记录整份文件或人物敏感详情。

### Review 中已修复

- 修复权限校验失败前已解析并返回人物姓名的跨支派信息暴露风险。
- 新增无权限预览不泄露姓名的回归测试。

## 5. Performance

### 已确认

- 列表、失败行分页和审核任务继续使用现有服务端分页。
- 通用草稿目标增加部分索引，仅索引非空业务草稿。
- 文件解析流式读取 CSV；XLSX 使用现有 POI 解析方式。
- 预览中的文件内重复使用 Set，文件内循环检测使用邻接表和 BFS。

### 非阻断风险

- 当前人物编码按行查询，适合当前同步导入规模；大批次场景会产生 N+1 查询。
- 超大批次异步化和批量预取属于已规划 Issue #107，本 PR 不提前引入复杂批处理。
