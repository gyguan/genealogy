# Issue #107 执行看板：超大导入批次异步处理与失败恢复

- Issue：https://github.com/gyguan/genealogy/issues/107
- 分支：`agent/issue-107-async-import-recovery`
- 目标：在复用 `import_job` / `import_job_row` 的前提下，建立可恢复、可观测、可取消的异步导入与分段发布机制，同时保留小批次同步体验。

## 方案与影响

- 大批次按阈值路由到异步执行，小批次继续同步处理。
- `import_job` 增加执行模式、阶段、游标、进度、重试、失败摘要、租约和取消/暂停状态；不新增第二套任务主模型。
- 分片使用稳定幂等键，事务边界收敛到单分片。
- 审核通过后的正式化进入 `publishing`，按分片提交并可从最近安全游标恢复。
- 前端任务中心展示阶段、进度、剩余量、失败摘要，并提供暂停、继续、取消、重试动作。

## 兼容与回滚

- 新字段提供默认值，历史任务按同步模式兼容。
- 新接口为增量接口，不删除现有上传、预览、重试和审核接口。
- 回滚代码后新增字段保留，不影响旧逻辑；数据库仅做前向补偿，不修改历史迁移。
- 任务分片通过幂等键和已提交游标防止重复草稿或重复正式化。

## 已知风险

- 多实例并发领取任务时必须使用数据库原子租约，不能只依赖进程内锁。

- 暂停和取消只在安全分片边界生效，不能中断正在提交的事务。
- 预计剩余量只能基于已处理速率估算，缺少样本时返回空值而非虚构。
