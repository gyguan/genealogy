# Issue #107 执行看板：超大导入批次异步处理与失败恢复

- Issue：https://github.com/gyguan/genealogy/issues/107
- 分支：`agent/issue-107-async-import-recovery`
- PR：https://github.com/gyguan/genealogy/pull/177
- 目标：在复用 `import_job` / `import_job_row` 的前提下，建立可恢复、可观测、可取消的异步导入与分段发布机制，同时保留小批次同步体验。

## 方案与影响

- 同时按文件大小和数据行数判定大批次；任一达到阈值即异步，小批次继续同步。
- `import_job` 增加执行模式、阶段、游标、进度、重试、失败摘要、租约和取消/暂停状态；不新增第二套任务主模型。
- 文件载荷独立持久化，上传请求完成表头/权限校验后快速返回任务编号。
- 多实例通过 PostgreSQL `FOR UPDATE SKIP LOCKED` 和过期租约领取任务。
- 每个草稿/发布分片使用稳定幂等键；人物草稿按原始行号防重复，正式化按 `published_at` 防重复。
- 审核通过后的异步批次进入 `publishing`，按分片事务提交并可从最近安全点恢复。
- 前端展示阶段、进度、剩余条数、分片大小、心跳、重试和失败摘要，并提供暂停、继续、取消、重试动作。

## 兼容与回滚

- 新字段提供默认值，历史任务按同步完成模式兼容。
- 新接口为增量接口，不删除现有上传、预览、失败行重试和审核接口。
- 回滚代码后新增字段和检查点保留，不影响旧同步逻辑；数据库仅做前向补偿，不修改历史迁移。
- 暂停和取消仅在安全分片边界生效，不中断正在提交的事务。
- 解析失败保留原始载荷；发布阶段不依赖原始文件，可从未发布行继续。

## 五轴 Review 自检

- Correctness：分片、行号、唯一幂等键和发布检查点共同避免重复或跳行；等待最终测试确认。
- Readability：执行状态、调度、人物分片、发布分片和控制接口按职责拆分。
- Architecture：复用通用导入任务/失败行/审核模型；后台执行不依赖 HTTP 生命周期。
- Security：所有进度与控制接口执行登录、宗族和支派写范围校验；错误摘要限制长度，不回传原始载荷。
- Performance：同步/异步双阈值路由、单分片事务、调度索引和分页展示；原始文件仅在后台解析阶段读取。

## 已知边界

- 当前异步解析首先覆盖人物导入；关系、来源和后续导入类型可复用同一执行/发布框架接入。
- CSV/XLSX 原始载荷当前持久化到数据库，后续达到对象存储规模时应切换为受控文件存储引用。
- “预计剩余量”当前按剩余数据条数展示，不在样本不足时虚构时间估算。
