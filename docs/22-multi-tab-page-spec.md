# 22. 多 TAB 页面设计与实现规范

> 本规范用于统一 Genealogy 中具有多个并列业务域的管理页面，重点约束页面结构、Tab 导航、查询 Card、结果 Card、操作位置、URL 状态、异步加载、响应式和工程实现。
>
> 本规范是 `docs/10-frontend-design-guidelines.md` 与 `docs/21-frontend-page-pattern-spec.md` 的专项补充。发生冲突时，依次遵循根目录 `AGENTS.md`、已批准的 Issue/Spec、前端目录级 `AGENTS.md`、本规范和通用页面规范。

## 1. 目标

多 TAB 页面必须解决以下问题：

1. 同一模块中的多个并列业务域保持统一结构和操作习惯；
2. Tab 切换后，标题、筛选条件、结果列表和主操作能够准确对应当前业务域；
3. 刷新、前进、后退和分享链接可以恢复当前 Tab 与查询现场；
4. 单个 Tab 加载或操作失败时，不影响其他 Tab；
5. 桌面端和移动端使用同一信息架构，不依赖横向滚动完成核心任务；
6. 页面可以通过配置复用公共骨架，但不把不同领域强行抽象为一个超大组件。

## 2. 适用范围

适用于以下页面：

- 同一模块下存在 `2～5` 个并列业务域；
- 各业务域共享页面上下文、宗族范围和权限边界；
- 用户需要频繁切换、查询和对照不同类型的数据；
- 每个 Tab 都有相对独立的查询条件、结果列表、详情和新增入口；
- 每个 Tab 可以独立加载、独立报错和独立维护 URL 状态。

典型页面：

- 宗族文化：文化资料、迁徙脉络、文化场所；
- 审核中心：待我审核、我提交的、已处理；
- 来源资料库：来源资料、附件资源、引用关系；
- 审计追踪：操作日志、审核流程、字段变更。

## 3. 不适用场景

以下情况不得使用页面级 Tabs：

- 内容存在明确先后顺序，应使用 `Steps`；
- 各分区属于不同模块、不同权限域或不同核心任务，应使用菜单或独立路由；
- 只有一个有效分区，应直接使用 Card 或普通页面；
- 仅为了缩短页面而拆分字段，应优先重新分组或使用锚点；
- 需要两层及以上嵌套 Tabs，应重新设计信息架构；
- Tab 数量超过 `7` 个，应重新归类、使用二级导航或拆分页面。

## 4. 标准页面结构

### 4.1 查询列表型多 TAB 页面

统一采用“查询 Card + 结果 Card”的双 Card 结构：

```text
页面上下文（由应用壳提供）

查询 Card
├─ Card 标题：模块名称
├─ Tabs：并列业务域
├─ 当前 Tab 常用查询条件
├─ 更多筛选（可展开）
└─ 操作区：更多筛选、重置、查询

结果 Card
├─ Card 标题：当前 Tab 对象名称 + 总数
├─ Card 右上角：当前 Tab 唯一主操作
├─ 局部告警、加载态、空态或错误态
├─ Table / 移动端 Record Card
└─ 分页

详情 Drawer / 独立编辑页
```

强制要求：

- 不额外增加只包含标题和说明的“介绍 Card”；
- 模块说明如确有必要，应放在查询 Card 标题附近，以次要文字展示；
- Tabs 放在查询 Card 内，切换后查询区域和结果区域同步变化；
- 查询 Card 与结果 Card 间距统一为 `16 px`；
- 查询条件之间不使用分隔线；
- “更多筛选”展开区域与常用条件保持连续表单布局，不增加顶部或底部分隔线。

### 4.2 详情型多 TAB 页面

详情页或详情 Drawer 可使用以下结构：

```text
Header：对象名称、状态、更新时间、高频操作
Tabs：基本信息、来源附件、审核追踪
当前 Tab 内容
```

详情型 Tabs 不得与查询列表型 Tabs 混为一层。列表页面打开详情后，详情 Tabs 应位于 Drawer 或独立详情页内部。

## 5. Tab 设计规范

### 5.1 数量与命名

- 推荐 `2～5` 个 Tab；
- 桌面端直接展示不得超过 `7` 个；
- Tab 名称使用 `2～6` 个汉字的稳定业务名词；
- 禁止使用“其他”“更多内容”“Tab1”等模糊名称；
- Tab 顺序按业务使用频率或稳定业务顺序确定，不按接口返回顺序动态变化；
- 无权限访问的业务域直接隐藏，不保留空壳 Tab；
- 暂时不可用但用户需要感知的 Tab 可以禁用，并提供原因说明。

### 5.2 选中状态

- 当前 Tab 必须同时具备颜色、下划线或底色等明确选中反馈；
- 不得只依赖颜色区分选中状态；
- Tab 点击热区桌面端不低于 `32 px`，移动端不低于 `44 px`；
- Tab 切换后焦点和页面滚动位置应保持可预测，不自动滚动到页面底部。

### 5.3 数量徽标

仅在数量具有明确业务价值时展示：

- 待办数量；
- 未读数量；
- 当前查询对象总数。

默认不显示 `0` 徽标，不使用高饱和颜色装饰普通总数。

### 5.4 主操作

每个激活 Tab 只允许一个视觉主操作，例如：

| Tab | 主操作 |
|---|---|
| 文化资料 | 新增文化资料 |
| 迁徙脉络 | 新增迁徙事件 |
| 文化场所 | 新增文化场所 |

主操作放在结果 Card 标题右侧，不在页面头、查询区和空态中重复展示同一按钮。首次空态可提供一次引导入口，但不得形成两个同级主按钮。

## 6. 查询 Card 规范

### 6.1 标题与 Tabs

查询 Card 标题使用模块名称，例如“宗族文化”。Tabs 紧随标题区域展示，形成统一的模块导航和查询入口。

不推荐：

```text
说明 Card：宗族文化 + 一段描述
Tabs Card：文化资料 / 迁徙脉络 / 文化场所
```

推荐：

```text
查询 Card：宗族文化
  Tabs：文化资料 / 迁徙脉络 / 文化场所
  查询条件
```

### 6.2 常用条件

- 使用 `Form layout="vertical"`；
- 桌面端默认一行展示 `4` 个查询条件；
- 常用条件优先展示，低频条件进入“更多筛选”；
- 字段顺序保持“范围 → 分类 → 细分范围 → 关键词/对象名称”；
- 不因文案较长缩小字号，应调整列宽或占用更多栅格；
- 查询条件之间不放竖线、横线或独立小 Card。

宗族文化标准首行：

```text
宗族 | 分类 | 支派 | 关键词
```

### 6.3 更多筛选

“更多筛选”用于承载低频但有明确业务价值的条件：

- 默认收起；
- 使用轻量文字按钮或 `Collapse ghost`；
- 展开后条件与首行表单保持相同栅格、标签和控件高度；
- 展开区域不增加分隔线；
- 展开状态可以保留在页面状态中，但无需写入 URL；
- 收起时不得清空已经生效的更多筛选条件；
- 存在已生效的隐藏条件时，应在“更多筛选”旁展示数量提示，例如“更多筛选（2）”。

宗族文化更多筛选：

```text
状态 | 可见范围 | 已有来源 | 首页精选
```

### 6.4 下拉选择

对于存在多个可选值的查询下拉：

- 必须支持多选；
- 必须支持清空；
- 选项较多时必须支持搜索；
- 必须提供“全选”能力；
- 选中项较多时使用 `maxTagCount="responsive"` 或“已选 N 项”，不得无限撑高控件；
- “全选”仅作用于当前可选范围；搜索过滤后执行全选时，必须明确是“全选全部”还是“全选当前搜索结果”；
- 选项来自接口时，加载失败应在控件附近提示，不得静默显示为空列表。

推荐实现方式：

1. 下拉顶部提供“全选 / 清空”；或
2. 使用明确的“全部”伪选项，并在状态模型中转换为所有有效值。

禁止同时提交“全部”和值列表，避免后端查询语义不一致。

### 6.5 关键词

- placeholder 应说明可查询的业务字段，例如“标题、摘要、时期或地点”；
- 查询前执行 `trim()`；
- 空字符串按未填写处理；
- 不在每次输入时自动请求，除非页面明确采用实时搜索且具备防抖、取消请求和加载反馈。

### 6.6 查询操作区

操作区位于查询 Card 右下角，顺序固定为：

```text
更多筛选 | 重置 | 查询
```

其中：

- “查询”为主操作；
- “重置”为默认按钮；
- 查询中保持按钮宽度不变并显示 loading；
- 查询成功不弹成功消息；
- 查询失败在结果 Card 内展示，不覆盖查询条件；
- 按 Enter 可触发查询，但多行文本和下拉搜索框除外。

### 6.7 重置语义

重置当前 Tab 时：

- 清空当前 Tab 查询条件；
- 恢复当前 Tab 默认排序和默认分页大小；
- 页码重置为 `1`；
- 关闭当前详情；
- 不切换 Tab；
- 不清空其他 Tab 已保存的查询状态；
- 不改变全局用户、组织或工作空间上下文。

## 7. 结果 Card 规范

### 7.1 标题

结果 Card 标题统一使用：

```text
对象名称（总数）
```

例如：

```text
文化资料（128）
迁徙事件（36）
文化场所（18）
```

数据未加载完成前，不使用 `0` 冒充真实数量。

### 7.2 表格

- 名称列放在最左侧，并作为进入详情的主入口；
- 操作列固定在最右侧；
- 操作列默认不超过“查看、编辑、更多”；
- 审核、归档、删除等低频或高风险动作进入“更多”；
- 操作列宽度保持稳定，避免切换 Tab 后频繁跳动；
- 列表使用后端分页，不在前端截取模拟分页；
- 排序、分页和查询条件共同构成当前 Tab 查询状态；
- 横向滚动只允许作为桌面端补偿方案，不得作为移动端核心交互。

### 7.3 状态展示

- 状态、可信度、可见范围等枚举使用 `Tag`；
- 展示中文业务文案，不展示原始枚举值；
- 证据类字段优先展示业务摘要，例如“来源 2 · 附件 3”；
- 时间字段统一展示为北京时间，并使用“最近更新”等业务名称。

### 7.4 加载、空态和错误态

每个 Tab 独立处理以下状态：

| 状态 | 展示要求 |
|---|---|
| 首次加载 | Table loading 或 Skeleton |
| 查询为空 | Empty + 重置筛选或新增入口 |
| 首次失败 | Result/Alert + 重新加载 |
| 刷新失败 | 保留上次成功结果，并提示数据可能不是最新 |
| 无权限 | 403，不展示受限对象名称、数量或摘要 |
| 对象不存在 | 404，关闭无效详情入口或提供返回列表 |

单个 Tab 的错误不得阻止用户切换到其他 Tab。

## 8. Tab 状态与 URL

### 8.1 URL 基线

激活 Tab 必须写入 URL：

```text
?view=<module>&tab=<tab-key>
```

要求：

- 刷新后恢复当前 Tab；
- 浏览器前进、后退恢复当前 Tab；
- 分享链接直接进入指定 Tab；
- 非法或过期 Tab key 自动回退默认 Tab，并通过 `replaceState` 规范化 URL；
- Tab key 使用稳定英文标识，不使用中文标题作为 URL 值。

### 8.2 查询状态

需要刷新和返回恢复的状态应写入 URL：

- 当前 Tab；
- 关键词；
- 多选筛选；
- 排序；
- 页码和分页大小；
- 当前打开的详情对象标识（按业务需要）。

不建议写入 URL：

- loading；
- 错误文案；
- 临时弹窗开关；
- 下拉展开状态；
- 未提交的敏感表单内容。

### 8.3 各 Tab 独立记忆

切换 Tab 时：

- 当前 Tab 的筛选、分页、排序和滚动位置应被保留；
- 返回原 Tab 时恢复原查询现场；
- 不得把 A Tab 的筛选字段错误带入 B Tab；
- 公共字段可以共享控件，但必须由各 Tab 状态显式映射；
- URL 只表达当前激活 Tab 的状态，其他 Tab 的临时状态可保存在页面级状态或缓存模型中。

## 9. 数据加载与并发控制

### 9.1 默认加载策略

- 默认只挂载和加载当前激活 Tab；
- 未访问 Tab 不提前请求业务列表；
- 用户首次切换到某 Tab 时再加载；
- 对切换频繁且数据量较小的场景，可以缓存最近成功结果；
- 缓存结果必须有明确失效或刷新机制。

### 9.2 请求隔离

每个 Tab 必须具备独立请求状态：

```text
loading / data / error / forbidden / lastSuccessAt
```

快速切换、重复查询或修改筛选时：

- 取消旧请求，或使用 request id 忽略过期响应；
- 旧请求不得覆盖新查询结果；
- 切换 Tab 后，原 Tab 的错误消息不得显示在新 Tab；
- Drawer 详情请求与列表请求独立管理。

### 9.3 刷新策略

新增、编辑、审核或归档成功后：

- 刷新当前 Tab 列表；
- 保留当前筛选；
- 保持合理页码；删除当前页最后一条时可回退上一页；
- 其他 Tab 只有在数据确实受影响时才标记为待刷新；
- 不无差别刷新所有 Tab。

## 10. 权限与操作

- 权限以后端返回的 `allowedActions` 或等价字段为准；
- 不使用一份通用权限判断覆盖所有行；
- 无权限访问的 Tab 不请求数据；
- 无权执行且无法恢复的操作直接隐藏；
- 因状态暂不可执行的操作可以禁用并说明原因；
- 删除、归档、撤销等风险操作必须确认影响对象；
- 前端权限展示不得替代后端鉴权；
- 不在 URL、表格或错误信息中暴露不必要的技术 ID 和敏感数据。

## 11. 响应式规范

### 11.1 桌面端

- 页面内容最大宽度遵循全局规范；
- 查询首行默认 `4` 个条件；
- Card 间距 `16 px`；
- 表格操作列固定右侧；
- Tabs 单行展示。

### 11.2 平板端

- 查询条件调整为每行 `2` 个；
- 操作区保持右对齐；
- 表格可以保留必要横向滚动，但名称和操作入口必须可达。

### 11.3 移动端

- 查询条件单列展示；
- Tabs 单行横向滚动，不允许换成两行；
- 首屏建议可见 `3～4` 个 Tab；
- 核心按钮高度 `40 px`、最小宽度 `88 px`；
- Table 转换为 Record Card 或块级行布局；
- 不允许依赖横向滚动查看名称、状态和主要操作；
- 详情 Drawer 占满视口；
- 禁止嵌套 Tabs。

## 12. 可访问性与键盘操作

- 使用 Ant Design `Tabs`、`Form`、`Table`、`Button` 等语义组件；
- Tab 必须具备正确的 `role`、`aria-selected` 和键盘切换能力；
- 图标按钮必须提供 Tooltip 和 `aria-label`；
- 名称链接和表格操作必须可通过键盘聚焦；
- 支持 Enter 打开详情，Esc 关闭 Drawer；
- 错误提示不能只依赖颜色；
- 异步更新应通过组件状态提供可感知反馈。

## 13. React + Ant Design 实现建议

### 13.1 目录结构

```text
features/<module>/
├─ <Module>ProductPage.tsx       # 页面壳、Tab 与公共上下文
├─ <DomainA>StandardTab.tsx      # A Tab 查询、列表和详情
├─ <DomainB>StandardTab.tsx      # B Tab 查询、列表和详情
├─ <DomainC>StandardTab.tsx      # C Tab 查询、列表和详情
├─ <module>PagePattern.ts        # Tab 名称、主操作、样式标识
├─ <module>TabState.ts           # Tab URL 状态
├─ <domain>UrlState.ts           # 各领域查询状态
├─ <module>Service.ts            # 领域服务
└─ <module>.css                  # 模块前缀样式
```

### 13.2 配置模型

推荐通过配置统一 Tab 元数据：

```ts
export type ManagedTabPattern = {
  key: string;
  label: string;
  resultTitle: string;
  primaryAction: string;
  mobileClass: string;
};
```

配置只承载稳定页面元数据，不承载复杂业务逻辑、权限推断或接口调用。

### 13.3 页面壳职责

页面壳只负责：

- 读取和写入当前 Tab；
- 加载公共上下文，例如宗族列表；
- 只挂载当前激活业务域；
- 处理浏览器前进和后退；
- 提供模块级消息容器。

页面壳不得：

- 聚合三个 Tab 的全部表单字段；
- 同时请求所有业务域数据；
- 通过大量 `if/else` 拼接不同领域表格；
- 推断领域权限和状态；
- 持有所有详情和编辑表单逻辑。

### 13.4 Tab 组件职责

每个 Tab 组件独立负责：

- 查询表单和查询状态；
- 列表请求、分页和排序；
- 当前领域详情；
- 当前领域新增、编辑和治理操作；
- 当前领域加载、错误和权限状态；
- 当前领域 URL 查询参数。

## 14. 宗族文化标准示例

### 14.1 查询 Card

```text
标题：宗族文化
Tabs：文化资料 | 迁徙脉络 | 文化场所

默认查询条件：
宗族 | 分类 | 支派 | 关键词

更多筛选：
状态 | 可见范围 | 已有来源 | 首页精选

操作：
更多筛选 | 重置 | 查询
```

要求：

- 所有适合多选的下拉支持多选、全选和清空；
- 查询条件之间不增加分隔线；
- 更多筛选展开后不增加分隔线；

- 宗族为公共上下文，但每个 Tab 的支派和筛选状态独立保存。

## 15. 禁止模式

禁止出现以下实现：

1. 独立说明 Card + Tabs Card + 查询 Card，造成三层重复容器；
2. Tab 放页面外，查询条件和结果标题却不随 Tab 变化；
3. 每个 Tab 重复页面级标题和相同主操作；
4. 查询条件之间使用分隔线或多个小 Card 切碎表单；
5. “更多筛选”展开后用粗分隔线划成另一块页面；
6. 多选下拉没有全选或清空能力；
7. 所有 Tab 首次进入时并发加载；
8. Tab 切换后丢失分页和筛选现场；
9. Tab 未写入 URL，刷新后总是回到第一个 Tab；
10. 使用 Tabs 模拟有先后关系的流程；
11. 桌面端表格直接缩小到移动端并要求横向滚动；
12. 页面壳包含所有领域请求、表格、详情和编辑逻辑；
13. 前端自行推断权限、审核状态或补造业务数据。
