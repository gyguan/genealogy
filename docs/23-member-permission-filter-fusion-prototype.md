# 成员与权限页面查询条件融合原型

## 1. 目标

将“成员”和“权限”两部分查询条件融合为一个统一筛选区，让用户先确定“查谁”，再补充“按什么权限范围过滤”，减少上下两组条件带来的重复阅读和操作跳转。

本原型仅调整交互布局和筛选信息架构，不改变成员授权、停用、撤销、审核留痕、后端权限校验和数据范围控制逻辑。

## 2. 现状问题

当前页面的查询入口容易被理解为两个独立查询：

- 成员维度：成员关键词、成员状态。
- 权限维度：角色、授权范围。

但实际列表结果是“成员 + 授权”的聚合结果，用户需要一次性组合条件定位成员。因此建议把两部分条件合并为同一张“成员与权限筛选”卡片。

## 4. 交互规则

1. 筛选区标题由“查询条件”调整为“成员与权限筛选”，明确这是同一个结果集的组合条件。
2. 输入顺序固定为：成员关键词 → 成员状态 → 角色 → 授权范围。
3. 成员关键词保留宽输入框，提示文案调整为“搜索成员姓名 / 账号”。
4. 筛选区底部增加“已选条件摘要”：
   - 无条件时显示“未设置筛选条件，将展示当前宗族内可见成员”。
   - 有条件时按“关键词、成员状态、角色、授权范围”的顺序展示。
5. “重置”只清空筛选条件并回到第一页，不改变当前宗族。
6. “查询”组合提交所有筛选条件，列表分页回到第一页。
7. 筛选条件继续同步到 URL，刷新和返回时可恢复当前查询。

## 5. 字段与文案

| 位置 | 文案 / 选项 | 说明 |
| --- | --- | --- |
| 卡片标题 | 成员与权限筛选 | 替代“查询条件” |
| 关键词占位 | 搜索成员姓名 / 账号 | 不要求用户输入技术 ID |
| 成员状态 | 全部状态 / 有效 / 已停用 / 已移除 | 使用业务文案 |
| 角色 | 全部角色 / 宗族管理员 / 支派管理员 / 采集员 / 查看成员 | 取后端可授权角色列表 |
| 授权范围 | 全部范围 / 全宗族 / 支派及下级支派 | 与后端范围枚举映射展示 |
| 空摘要 | 未设置筛选条件，将展示当前宗族内可见成员 | 说明后端仍执行权限范围控制 |

## 6. 验收点

- 成员与权限筛选条件位于同一张卡片内。
- 桌面端一行展示 4 个筛选项；窄屏按单列纵向排列。
- 筛选提交、重置和分页行为与现有列表查询一致。
- 不新增 API 字段，不改变 OpenAPI 契约。
- 不绕过后端鉴权、数据范围和成员授权审计逻辑。
