# 宗族文化页面筛选内聚原型

## 1. 背景

当前宗族文化页顶部存在全局说明文案、全局“当前宗族”选择器和全局新增按钮：

- 说明文案：`查询和维护宗族文化资料、迁徙脉络及文化场所。`
- 全局宗族选择：位于页面 Header 右侧。
- 全局新增按钮：根据当前 Tab 切换为新增资料、新增迁徙事件或新增场所。

本原型仅调整信息架构与交互位置，不变更领域模型、API 契约、权限判定、审核流和数据生效路径。

## 2. 设计目标

1. 删除页面顶部说明文案，降低首屏冗余信息。
2. 将“当前宗族”从页面全局 Header 移入各 Tab 的查询条件区域，让宗族成为当前查询上下文的一部分。
3. 将新增按钮移入各 Tab 内容区，靠近对应列表和筛选条件。
4. 保持每个 Tab 的查询、分页、空态、错误态、无权限态独立，便于后续按 Tab 拆分和维护。
5. 不改变后端权限语义：前端仍只消费后端返回的 `allowedActions` 或新增权限聚合结果，不能替代后端鉴权。

## 3. 总体布局原型

### 3.1 页面 Header

```text
┌──────────────────────────────────────────────────────────────┐
│ 宗族文化                                                     │
│ ┌──────────┬────────────┬──────────┐                         │
│ │ 文化资料 │ 迁徙脉络   │ 文化场所 │                         │
│ └──────────┴────────────┴──────────┘                         │
└──────────────────────────────────────────────────────────────┘
```

调整点：

- 保留页面标题“宗族文化”。
- 删除标题下方说明文案。
- 删除 Header 右侧的“当前宗族”选择器。
- 删除 Header 右侧的全局新增按钮。
- Header 只承担页面识别与 Tab 导航，不承担业务查询条件。

## 4. Tab 内布局共性

每个 Tab 采用统一的“查询卡片 + 列表卡片”结构：

```text
┌──────────────────────────────────────────────────────────────┐
│ 查询条件                                                     │
│ 宗族 [请选择宗族 v]  关键词/分类/支派/状态...                │
│ 更多筛选                                                     │
│                                      [重置] [查询] [新增...] │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│ 列表标题（总数）                                             │
│ 表格 / 时间线 / 卡片列表 / 空态 / 错误态 / 无权限态          │
└──────────────────────────────────────────────────────────────┘
```

共性规则：

- 宗族选择是每个 Tab 查询区的第一项。
- “新增”按钮位于查询卡片右下角或列表卡片右上角，推荐放在查询卡片右下角，与“查询”形成同一操作带。
- 未选择宗族时：
  - 查询按钮禁用或仅允许选择宗族后查询；
  - 新增按钮禁用；
  - 列表区域显示“请选择宗族后查看……”空态。
- 切换宗族时：
  - 清空当前 Tab 的支派筛选；
  - 重置页码到第 1 页；
  - 关闭当前详情抽屉和编辑器；
  - 不影响其他 Tab 已保存在 URL 中的筛选条件，除非这些条件依赖旧宗族。

## 5. 文化资料 Tab 原型

```text
┌──────────────────────────────────────────────────────────────┐
│ 文化资料查询                                                 │
│ 宗族      [王氏宗族 v]                                       │
│ 关键词    [标题、摘要、时期或地点           ]                 │
│ 分类      [可多选 v]     支派 [可多选 v]     状态 [可多选 v] │
│ 更多筛选：可见范围 / 已有来源 / 首页精选 / 排序              │
│                                      [重置] [查询] [新增资料] │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│ 文化资料（128）                                              │
│ 标题 | 分类 | 所属范围 | 可信度 | 状态 | 可见范围 | 证据 | 操作 │
└──────────────────────────────────────────────────────────────┘
```

交互说明：

- “宗族”选择复用当前宗族列表接口和展示名称规则。
- 文化资料查询条件保持现有能力：关键词、分类、支派、状态、可见范围、已有来源、首页精选、排序。
- 新增资料按钮只在已选择宗族时可用。
- 行内编辑、提交审核、归档、删除继续基于单条数据的 `allowedActions` 展示。
- 首次进入页面时，如果工作区已有宗族，则默认填入；如果没有宗族，可选择保持当前自动选第一个宗族的行为，或改为强制用户选择，建议后续实现前单独确认。

## 6. 迁徙脉络 Tab 原型

```text
┌──────────────────────────────────────────────────────────────┐
│ 迁徙事件查询                                                 │
│ 宗族      [王氏宗族 v]                                       │
│ 支派      [请选择支派 v]   关键词 [迁出地、迁入地、始迁祖]    │
│ 状态      [可多选 v]       可信度 [可多选 v]                 │
│ 更多筛选：可见范围 / 敏感级别 / 时间文本 / 来源覆盖          │
│                                  [重置] [查询] [新增迁徙事件] │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│ 迁徙脉络（按支派和迁徙序号展示）                             │
│ 时间线 / 表格 / 空态 / 无权限态                              │
└──────────────────────────────────────────────────────────────┘
```

交互说明：

- 迁徙事件领域要求 `branch_id` 必填；因此新增迁徙事件时，表单内仍必须选择明确支派。
- 查询区的宗族变更后，需要重新加载支派选项，并清空旧支派条件。
- 如果未选择宗族，列表显示“请选择宗族后查看迁徙脉络”。

## 7. 文化场所 Tab 原型

```text
┌──────────────────────────────────────────────────────────────┐
│ 文化场所查询                                                 │
│ 宗族      [王氏宗族 v]                                       │
│ 关键词    [名称、地点、说明                  ]                │
│ 类型      [祠堂/祖居/墓园/纪念设施 v]  支派 [可多选 v]       │
│ 状态      [可多选 v]       现实状态 [现存/重建/迁建/损毁]    │
│ 更多筛选：可见范围 / 敏感级别 / 是否有坐标 / 来源覆盖        │
│                                      [重置] [查询] [新增场所] │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│ 文化场所（36）                                               │
│ 名称 | 类型 | 所属范围 | 地点 | 现实状态 | 状态 | 可见范围 | 操作 │
└──────────────────────────────────────────────────────────────┘
```

交互说明：

- 文化场所涉及祠堂、祖居、墓园、纪念设施等地点型数据。
- 对私人住宅、墓地或封存地点，前端仅展示后端裁剪后的地点、坐标和附件信息。
- 新增场所按钮只在已选择宗族时可用。

## 8. URL 状态建议

推荐将宗族作为各 Tab URL 状态的一部分，而不是只依赖全局工作区状态：

```text
/culture?tab=items&clanId=1&keyword=...&category=hall_name&pageNo=1
/culture?tab=migrations&clanId=1&branchId=12&pageNo=1
/culture?tab=sites&clanId=1&siteType=ancestral_hall&pageNo=1
```

建议规则：

1. 页面进入时优先读取 URL 中的 `clanId`。
2. URL 没有 `clanId` 时，再使用 `workspace.clanId`。
3. 两者都没有时，由产品确认是自动选择第一个宗族，还是要求用户手动选择。
4. 用户在任一 Tab 修改宗族后，同步更新 `workspace.clanId` 和当前 Tab URL。
5. 切换 Tab 时，如果目标 Tab 没有自己的 `clanId`，可继承 `workspace.clanId`。

## 9. 推荐方案

推荐采用“Header 极简 + Tab 内查询内聚”的方案：

- Header：只保留标题和 Tab。
- 各 Tab 查询卡片：内置宗族选择、业务筛选、重置、查询和新增按钮。
- 各 Tab 列表卡片：负责列表、空态、错误态、无权限态和详情入口。

该方案的好处：

1. 页面顶部更轻，减少用户在 Header 和查询区之间来回寻找操作。
2. 宗族选择与筛选条件在同一区域，符合“当前查询上下文”的直觉。
3. 新增按钮靠近对应业务列表，减少全局按钮随 Tab 切换带来的认知负担。
4. 后续如果三类 Tab 的权限、筛选和创建入口差异扩大，也更容易独立演进。

## 10. 待确认问题

实现前建议确认以下产品选择：

1. 未选择宗族时，是否继续自动选择第一个可访问宗族？
   - 推荐：暂时保留现有自动选择，降低使用门槛；如果担心误操作，再改为手动选择。
2. “新增”按钮最终位置放在查询卡片右下角，还是列表卡片右上角？
   - 推荐：查询卡片右下角，与查询动作同一操作带。
3. 各 Tab 是否共享同一个宗族选择？
   - 推荐：视觉上在各 Tab 内选择，状态上仍同步到 `workspace.clanId`，切换 Tab 时继承最近选择。
