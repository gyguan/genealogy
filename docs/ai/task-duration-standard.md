# 执行看板、状态与耗时规范

本文档是仓库中**执行任务看板、任务状态、活跃耗时和耗时汇总的唯一权威口径**。

适用于：

- 聊天式长任务；
- Issue 实现与恢复；
- Draft PR 任务看板；
- `tasks/issue-<N>-execution.md`；
- CI 修复和 Review 整改；
- 进度汇报和最终交付摘要。

其他文档和 PR 模板可以引用本规范，原则上不再重复定义完整看板格式、状态语义和耗时口径。

---

## 1. 标准看板格式

所有新建执行任务看板必须使用以下格式：

```markdown
| 序号 | 任务 | 状态 | 耗时 | Commit / 结果或说明 |
|---|---|---|---|---|
| 1 | 读取规则与需求 | ✅ 已完成 | 约 4 分钟 | 已确认范围 |
| 2 | 实现分页查询 | 🔄 进行中 | 已累计约 12 分钟 | 正在补测试 |
| 3 | 执行完整验证 | ⏳ 待处理 | — |  |
```

在聊天中展示时，最后一列可以简化为“结果或说明”；在 PR 和任务文件中建议使用“Commit / 结果或说明”。

看板必须同步维护在：

1. `tasks/issue-<N>-execution.md`；
2. Draft PR 描述；
3. 必要的聊天进度更新；
4. 最终交付摘要。

---

## 2. 任务状态

| 标识 | 状态 | 使用条件 | 耗时填写 |
|---|---|---|---|
| ✅ | 已完成 | 已产生结果，并完成必要验证 | 固化累计活跃耗时 |
| 🔄 | 进行中 | 当前执行中，或已修改但尚未验证 | `已累计约 N 分钟` |
| ⏳ | 待处理 | 尚未开始 | `—` |
| ⚠️ | 阻塞 | 因权限、信息、依赖、CI 或技术问题无法继续 | 保留阻塞前累计耗时 |
| ❌ | 失败 | 执行失败且尚未修复 | 记录失败前累计耗时 |
| ⏭️ | 已跳过 | 确认无需执行，并说明原因 | 未执行填 `—`；已分析则记录分析耗时 |

原则上同一时间只保留一个主要任务为“🔄 进行中”。

代码已修改但未提交、未检查 diff 或未完成必要验证时，不得标记为“✅ 已完成”。

---

## 3. 任务粒度

任务必须描述可验证结果，不使用无法判断完成条件的笼统描述。

不推荐：

- 修改后端；
- 优化前端；
- 完善测试。

推荐：

- 增加来源资料分页查询接口；
- 增加标题和来源类型筛选条件；
- 修改列表页筛选区域；
- 增加详情页引用记录查询；
- 补充搜索接口单元测试；
- 修复前端类型检查失败。

每个任务应满足：

1. 有明确输入与输出；
2. 可以独立判断是否完成；
3. 尽量对应具体文件或提交；
4. 有验收标准与验证方式；
5. 规模足够小，便于 Review、计时和恢复。

---

## 4. 耗时口径

“耗时”默认指**实际活跃执行耗时**。

计入：

- 阅读直接相关规则、需求、代码和文档；
- 方案分析、任务拆解和影响评估；
- 代码、文档、配置和测试修改；
- diff 检查、Review 和问题修复；
- 发起验证、读取结果和定位失败原因；
- 更新任务文件、PR 和恢复检查点。

不计入：

- 等待用户确认、授权或补充信息；
- GitHub Actions 排队和自动运行时间；
- 等待外部系统、网络或第三方服务；
- 会话中断、页面关闭或明确暂停；
- 与当前任务无关的工作。

外部等待对交付有明显影响时，应在“结果或说明”中单独记录，例如：

```text
活跃耗时约 8 分钟；CI 排队与运行等待 6 分钟（不计入耗时）
```

---

## 5. 耗时展示

| 场景 | 写法 |
|---|---|
| 少于 1 分钟 | `<1 分钟` |
| 1～59 分钟 | `约 N 分钟` |
| 60 分钟及以上 | `约 H 小时 M 分钟` |
| 进行中 | `已累计约 N 分钟` |
| 待处理 | `—` |
| 规范生效前无法还原 | `未记录（规范生效前）` |

除非由系统自动生成，否则不记录秒级精度。不得使用无来源的精确值、未来预计值或事后猜测值。

---

## 6. 计时生命周期

### 6.1 开始任务

任务从“待处理”切换为“进行中”时：

1. 记录首次开始时间，使用北京时间；
2. 开始累计活跃执行耗时；
3. 同一时间原则上只计时一个主要任务。

### 6.2 暂停、等待或阻塞

出现以下情况时暂停累计：

- 等待用户决策；
- 等待权限或外部依赖；
- 等待 CI；
- 会话准备中断；
- 切换到另一主要任务。

暂停时保存当前累计耗时，并在说明中写明原因。

### 6.3 恢复任务

恢复后：

1. 读取任务文件和 PR 中最后记录的累计耗时；
2. 从新的活跃区间继续累计；
3. 不把中断期间计入耗时；
4. 无法确认历史耗时时，使用 `未记录（规范生效前）`，不得补造。

### 6.4 完成任务

任务满足完成判定后：

1. 停止计时；
2. 固化累计耗时；
3. 同步任务文件、PR 看板和进度汇报；
4. 写明 Commit、验证和结果。

代码已修改但未完成必要验证时，任务仍为“进行中”，耗时继续累计到下一次活跃执行结束。

---

## 7. 更新频率

耗时至少在以下节点更新：

- 任务开始；
- 任务完成、失败、跳过或进入阻塞；
- 每次恢复检查点；
- 每次 PR 任务看板更新；
- 最终交付。

标准模式下，不要求每分钟刷新聊天消息；只需保证 GitHub 中的持久化记录在原子任务切换时准确更新。

---

## 8. 耗时汇总

PR 或最终交付可以增加汇总：

```markdown
### 耗时汇总

- 已完成任务活跃耗时：约 1 小时 18 分钟
- 当前进行中累计耗时：约 12 分钟
- 外部等待：CI 约 9 分钟，不计入活跃耗时
- 未记录历史任务：2 项
```

汇总规则：

- 只汇总有可靠记录的任务；
- “未记录”不得按 0 计算；
- 跳过且未执行的任务不计入；
- 外部等待与活跃耗时分开展示；
- 不根据 Commit 时间差直接推算活跃耗时。

---

## 9. 过渡规则

本规范生效前已经启动的 Issue 或 PR：

- 不要求追溯并猜测历史耗时；
- 已完成步骤填写 `未记录（规范生效前）`；
- 从下一次恢复或任务切换开始记录；
- 不得因为缺少历史耗时而篡改 Commit 时间或补造记录。

---

## 10. 禁止事项

禁止：

- 把会话持续时间直接当作任务耗时；
- 把 CI 排队和自动运行时间计入活跃执行耗时；
- 为了让数据好看而缩短、平滑或补造耗时；
- 用未来预计耗时替代实际已发生耗时；
- 将“未记录”填写为 `0 分钟`；
- 任务状态、耗时和 GitHub 实际结果不一致。

耗时数据用于复盘任务复杂度和流程瓶颈，不用于证明 AI 始终快速，也不应替代质量、风险和验收结果。
