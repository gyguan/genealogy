# Issue 交付耗时经验

本文沉淀近期 Issue 实现与收尾过程中观察到的耗时规律，用于后续分析需求、拆分 Issue、实现 Issue、继续 PR 和收尾 PR 时预判成本、控制范围和减少不必要动作。

## 0. 强制适用范围

自本文档生效后，后续创建、拆分、实现、继续和收尾 Issue 时，必须把本文作为 `docs/ai/issue-creation-standard.md` 与 `docs/ai/issue-execution-governance.md` 的补充门禁执行。

- 创建 Issue 时，必须先按本文的轻量 / 重型分级、拆分信号和轻契约 / 重契约规则判断范围，避免把 UI、后端治理、数据库、权限、审核、CI 和历史基线治理混成一个大 Issue。
- 实现 Issue 时，必须在 `tasks/issue-<N>-execution.md` 中记录所选 Issue 类型、流程强度、验证强度、是否需要拆分，以及活跃耗时 / 外部等待分离口径。
- 继续或收尾 Issue 时，必须优先引用已有自动门禁、diff、PR Review 和任务看板事实，不得为轻量任务补造重型 Review 或高成本专项验证。
- 如需偏离本文默认策略，必须在 Issue、任务看板或 PR 中说明原因、影响、风险和补偿验证。

## 1. 核心结论

近期耗时增加的主要原因不是单纯编码效率下降，而是需求类型从页面展示、文案、布局调整，进入了导入、审核、权限、隐私、数据库迁移、契约治理和跨模块状态机等治理闭环型需求。

后续默认判断：

- UI 文案、菜单、样式和单页面调整，按轻量需求处理；
- Contract First、数据库迁移、权限、隐私、审核、认证、生产准出和跨模块状态机，按重型需求处理；
- 重型需求优先拆 Issue，不与前端页面、文案体验、CI 修复和历史基线治理混在一个 Issue；
- 高风险验证默认只记录为手工验证项，用户明确要求后再手工触发专项测试、E2E 或完整五轴 Review。

## 2. 最耗时步骤排序

| 排名 | 耗时步骤 | 典型原因 | 默认处理策略 |
|---|---|---|---|
| 1 | 权限 / 隐私 / 审核闭环 | 需要判断谁能看、谁能改、是否审核、日志能否披露、无权限返回 forbidden 还是 not-found | 单独拆分治理 Issue，避免和页面实现混做 |
| 2 | Contract First / OpenAPI / 生成类型 | 路径、参数、响应、错误码、兼容窗口、masked、warnings、容量边界容易扩散 | 区分轻契约和重契约，默认只做必要契约 |
| 3 | 数据库迁移 / 索引 / 状态字段 | 要考虑历史数据、回滚、补偿迁移、索引和旧流程兼容 | 数据库变更单独评估，迁移与业务实现不要混成大包 |
| 4 | 状态机 / 幂等 / 并发安全 | 异步、租约、重试、暂停、取消、分片发布、版本冲突、部分成功都需要异常路径 | 先定义状态与恢复点，再实现最小闭环 |
| 5 | 前后端联动和页面状态恢复 | URL、筛选、分页、详情 Drawer、错误态、无权限态、刷新恢复会叠加复杂度 | 前端页面与后端能力分 Issue 或分 PR 推进 |
| 6 | 验证 / CI / Review / 收尾 | 全量测试、五轴 Review、临时 workflow、执行记录和 Issue 回写会放大收尾成本 | 默认轻量 VERIFY / REVIEW，只记录手工验证建议 |

## 3. Issue 类型耗时分级

| Issue 类型 | 建议目标活跃耗时 | 说明 |
|---|---:|---|
| 文案 / 菜单 / 样式 | 5～15 分钟 | 只做 diff、影响范围和现有自动门禁状态确认 |
| 单页面前端调整 | 10～30 分钟 | 以页面代码复核和前端构建门禁为主 |
| 单接口 / 单契约变更 | 20～40 分钟 | 只覆盖核心路径、参数、响应和错误码 |
| 后端 CRUD + 分页 + 基础权限 | 45～90 分钟 | 涉及 DTO、Service、Repository、Controller 和基础测试 |
| 导入 / 审核 / 权限 / 隐私闭环 | 90～180 分钟 | 必须拆清楚审核、生效、日志、权限和数据披露 |
| 跨模块状态机 / 异步任务 / 发布准出 | 2 小时以上 | 必须拆分，不能作为单个大 Issue 推进 |

这些时间只作为经验分级，用于判断是否需要拆分，不作为承诺时间或预计完成时间。

## 4. 后续压缩策略

### 4.1 压缩 PR / Issue 启动和收尾

轻量需求不需要过重流程。默认采用：

```text
创建 Issue / 分支 / PR
→ 最小任务看板 2～3 项
→ 实现
→ diff 检查
→ 合入或记录待合入状态
```

不为轻量需求强制展开完整恢复检查点、完整五轴 Review 或专项验证。

### 4.2 区分轻契约和重契约

轻契约只覆盖：

- 路径；
- 参数；
- 核心响应字段；
- 主要错误码；
- 是否影响生成类型。

重契约才覆盖：

- 兼容窗口；
- 权限投影；
- masked / warnings；
- 容量边界；
- 回滚方式；
- 侧信道防护。

只有世系图谱、权限、导入、审核、认证和生产准出等核心链路默认考虑重契约。

### 4.3 权限 / 隐私 / 审核只描述本次变化

后续 Issue 正文只写本次新增或改变的权限点、隐私点和审核点。不要每次重复展开完整权限矩阵、隐私矩阵和审核 apply 全流程。

### 4.4 CI 结果只引用自动门禁

默认只记录现有自动门禁结果：

```text
Backend CI / Frontend CI / API Contract / Database Migration Governance
```

专项测试、Auth E2E、完整五轴 Review 和人工验收只登记为手工建议项，用户明确要求后再执行。

### 4.5 活跃耗时和外部等待必须分离

任务看板、PR 描述和最终 Issue 回写必须分开记录：

- 活跃耗时：规则 / 现场恢复、代码实现、聚焦验证、diff 检查、PR / Issue 收尾；
- 外部等待：CI 排队、CI 运行、Review 等待、用户确认、第三方服务等待；
- 未可验证耗时：历史会话、跨会话完成但无任务记录、外部系统缺失日志。

不得用会话持续时间替代活跃耗时，不得把 CI 等待时间计入实现效率分析。无法验证的历史耗时必须标记为“未形成可验证累计值”，禁止事后补造。

### 4.6 复用测试 fixture 和收尾模板

高频领域必须沉淀可复用测试样例和收尾模板，减少每个 Issue 重复构造成本：

- Tree / 世系图谱：节点、边、masked 人物、多父、继嗣、婚配、环、孤立、容量截断、来源 / 审核摘要等 fixture；
- 来源 / 审核 / 权限：来源绑定、审核状态、权限省略、隐私最小披露和审计日志断言 helper；
- 前端状态治理：请求版本门、分页、宗族切换、跨支派一致性和旧响应丢弃测试 helper；
- PR 收尾：验证结果、diff 范围、风险、回滚、Issue 回写和耗时汇总使用固定模板。

新增同类 Issue 时，应优先复用既有 fixture / helper / 模板；确需新增重复样例时，必须说明原因。

## 5. 拆分经验

遇到以下信号时，优先拆 Issue：

- 同时涉及前端、后端、数据库和 OpenAPI；
- 同时涉及权限、隐私、审核、日志和追踪；
- 同时出现状态机、异步任务、重试、暂停、取消、恢复点；
- 需要新增 Flyway 迁移并改变业务状态字段；
- 需要兼容旧接口、旧数据或旧前端入口；
- 需要引入临时 workflow 或专项 CI；
- 单个 PR 的验收标准超过 8 条，或任务看板超过 8 个原子任务。

推荐拆分顺序：

```text
领域模型 / 数据结构
→ API 契约
→ 后端最小闭环
→ 权限 / 隐私 / 审核治理
→ 前端页面接入
→ 追踪 / 日志 / 导出等增强
→ 手工验证 / 生产准出建议
```

## 6. 重型 Issue 并行拆分策略

重型需求默认先做并行拆分，不直接进入单 Issue 实现。并行化目标是减少自然历时和等待时间，而不是在一个不可恢复的大 PR 中同时修改所有层。

推荐结构：

```text
EPIC Issue
  ├─ API 契约 / DTO 子 Issue
  ├─ 数据库 / 状态字段子 Issue
  ├─ 后端最小闭环子 Issue
  ├─ 权限 / 隐私 / 审核治理子 Issue
  ├─ 前端页面接入子 Issue
  └─ 文档 / 手工验证建议 / 收尾子 Issue
```

每个子 Issue 原则上使用独立分支、独立 Draft PR、独立任务看板和独立自动门禁。最后由主控任务或集成 PR 统一收口依赖、冲突、验收标准和 Issue 回写。

### 6.1 适合并行的工作

- 需求影响面扫描：前端、后端、数据库、API、权限可以分线分析；
- Contract First：API 契约稳定后，后端和前端可以基于契约并行准备；
- 后端非共享模块：不同 Service、Repository、Adapter 可拆成不同子 Issue；
- 前端页面接入：在契约稳定后可与后端最小闭环并行推进；
- 文档、执行记录、手工验证建议：可和代码实现并行维护；
- GitHub Actions：自动门禁天然并行，默认只引用已有自动结果。

### 6.2 必须串行收口的共享资产

以下内容必须由主控任务串行定版，避免多分支互相覆盖：

- Flyway 迁移版本号和数据库结构最终形态；
- OpenAPI 公共路径、字段命名、错误码和兼容窗口；
- 权限码、角色映射、隐私分级和审核状态机；
- 共享实体、核心 Service、公共 DTO 和前端全局状态模型；
- 最终合入 `main`、关闭 EPIC / 子 Issue 和发布说明。

### 6.3 ChatGPT 页面直连 GitHub 的执行边界

当前协作模式不能真正启动多个后台 Agent 同时编码。后续默认采用“主控 Agent”方式执行：先拆分并行子 Issue / Draft PR，明确依赖关系和可并行边界，再逐个推进最小任务，并利用 GitHub Issue、分支、PR、Actions 和 Review 承载并行状态。

如果用户要求“并行推进”，默认含义是：

```text
先拆分可并行子 Issue
→ 为每条子线明确输入输出和依赖
→ 分别建立远程分支 / Draft PR / 任务看板
→ 避免共享资产冲突
→ 最后由主控任务串行集成收口
```

不得承诺在 ChatGPT 页面中真实启动多个后台执行体；不得把尚未推进的子 Issue 描述为已经并行完成。

## 7. 默认执行口径

后续执行 Issue 时默认遵循：

- 不主动扩大 VERIFY / REVIEW；
- 不主动触发高成本 E2E；
- 不把手工验证项当作已验证结果；
- 不把历史基线治理混入业务需求；
- 不把 UI 调整和后端治理闭环混成一个不可恢复的大任务；
- 如果发现需求进入重型区，先建议拆分或标记手工验证项，再继续当前最小任务；
- 如果发现需求具备并行条件，优先拆成 EPIC + 子 Issue + 独立 Draft PR，再由主控任务串行集成。

## 8. 创建和实现 Issue 的必填落点

### 8.1 创建 Issue 必填

创建 Issue 正文必须体现本文判断结果：

- Issue 类型：文案 / 菜单 / 样式、单页面前端调整、单接口 / 单契约变更、后端 CRUD + 分页 + 基础权限、导入 / 审核 / 权限 / 隐私闭环、跨模块状态机 / 异步任务 / 发布准出，或自定义类型；
- 流程强度：轻量 / 标准 / 重型；
- 契约强度：不涉及 / 轻契约 / 重契约；
- 是否触发拆分信号；如触发，必须说明拆分后的子 Issue 或不拆分原因；
- 验证强度：自动门禁、聚焦测试、全量测试、专项 E2E、手工验证建议；
- 是否包含历史基线治理；如包含，必须说明为什么不能拆到独立 Issue。

### 8.2 实现 Issue 必填

`tasks/issue-<N>-execution.md` 必须包含以下字段或等价信息：

- 本 Issue 类型、流程强度、契约强度和验证强度；
- 是否命中拆分信号，以及实际拆分 / 不拆分结论；
- 活跃耗时与外部等待分离口径；
- 复用的测试 fixture、helper、模板或无法复用的原因；
- 轻量任务不得展开重型 Review；重型任务不得降级为轻量验证，除非 PR 中记录明确风险和补偿措施。

### 8.3 收尾必填

PR 和 Issue 最终摘要必须包含：

- 活跃耗时汇总；
- 外部等待说明；
- 实际执行的验证命令或自动门禁；
- 未执行的高成本验证及原因；
- 是否存在后续拆分 Issue、手工验证建议或已知基线问题。
