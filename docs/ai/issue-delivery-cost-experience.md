# Issue 交付耗时经验

本文沉淀近期 Issue 实现与收尾过程中观察到的耗时规律，用于后续分析需求、拆分 Issue、实现 Issue、继续 PR 和收尾 PR 时预判成本、控制范围和减少不必要动作。

## 1. 核心结论

近期耗时增加的主要原因不是单纯编码效率下降，而是需求类型从页面展示、文案、布局调整，进入了导入、审核、权限、隐私、数据库迁移、契约治理和跨模块状态机等治理闭环型需求。

后续默认判断：

- UI 文案、菜单、样式和单页面调整，按轻量需求处理；
- Contract First、数据库迁移、权限、隐私、审核、认证、生产准出和跨模块状态机，按重型需求处理；
- 重型需求优先拆 Issue，不与前端页面、文案体验、CI 修复和历史基线治理混在一个 Issue；
- 高风险验证默认只记录为手工验证项，用户明确要求后再手工触发专项测试、E2E 或完整五轴 Review。

## 2. 最耗时步骤排序

| 排名 | 耗时步骤 | 典型原因 | 默认处理策略 |
|---|---|---|---|
| 1 | 权限 / 隐私 / 审核闭环 | 需要判断谁能看、谁能改、是否审核、日志能否披露、无权限返回 forbidden 还是 not-found | 单独拆分治理 Issue，避免和页面实现混做 |
| 2 | Contract First / OpenAPI / 生成类型 | 路径、参数、响应、错误码、兼容窗口、masked、warnings、容量边界容易扩散 | 区分轻契约和重契约，默认只做必要契约 |
| 3 | 数据库迁移 / 索引 / 状态字段 | 要考虑历史数据、回滚、补偿迁移、索引和旧流程兼容 | 数据库变更单独评估，迁移与业务实现不要混成大包 |
| 4 | 状态机 / 幂等 / 并发安全 | 异步、租约、重试、暂停、取消、分片发布、版本冲突、部分成功都需要异常路径 | 先定义状态与恢复点，再实现最小闭环 |
| 5 | 前后端联动和页面状态恢复 | URL、筛选、分页、详情 Drawer、错误态、无权限态、刷新恢复会叠加复杂度 | 前端页面与后端能力分 Issue 或分 PR 推进 |
| 6 | 验证 / CI / Review / 收尾 | 全量测试、五轴 Review、临时 workflow、执行记录和 Issue 回写会放大收尾成本 | 默认轻量 VERIFY / REVIEW，只记录手工验证建议 |

## 3. Issue 类型耗时分级

| Issue 类型 | 建议目标活跃耗时 | 说明 |
|---|---:|---|
| 文案 / 菜单 / 样式 | 5～15 分钟 | 只做 diff、影响范围和现有自动门禁状态确认 |
| 单页面前端调整 | 10～30 分钟 | 以页面代码复核和前端构建门禁为主 |
| 单接口 / 单契约变更 | 20～40 分钟 | 只覆盖核心路径、参数、响应和错误码 |
| 后端 CRUD + 分页 + 基础权限 | 45～90 分钟 | 涉及 DTO、Service、Repository、Controller 和基础测试 |
| 导入 / 审核 / 权限 / 隐私闭环 | 90～180 分钟 | 必须拆清楚审核、生效、日志、权限和数据披露 |
| 跨模块状态机 / 异步任务 / 发布准出 | 2 小时以上 | 必须拆分，不能作为单个大 Issue 推进 |

这些时间只作为经验分级，用于判断是否需要拆分，不作为承诺时间或预计完成时间。

## 4. 后续压缩策略

### 4.1 压缩 PR / Issue 启动和收尾

轻量需求不需要过重流程。默认采用：

```text
创建 Issue / 分支 / PR
→ 最小任务看板 2～3 项
→ 实现
→ diff 检查
→ 合入或记录待合入状态
```

不为轻量需求强制展开完整恢复检查点、完整五轴 Review 或专项验证。

### 4.2 区分轻契约和重契约

轻契约只覆盖：

- 路径；
- 参数；
- 核心响应字段；
- 主要错误码；
- 是否影响生成类型。

重契约才覆盖：

- 兼容窗口；
- 权限投影；
- masked / warnings；
- 容量边界；
- 回滚方式；
- 侧信道防护。

只有世系图谱、权限、导入、审核、认证和生产准出等核心链路默认考虑重契约。

### 4.3 权限 / 隐私 / 审核只描述本次变化

后续 Issue 正文只写本次新增或改变的权限点、隐私点和审核点。不要每次重复展开完整权限矩阵、隐私矩阵和审核 apply 全流程。

### 4.4 CI 结果只引用自动门禁

默认只记录现有自动门禁结果：

```text
Backend CI / Frontend CI / API Contract / Database Migration Governance
```

专项测试、Auth E2E、完整五轴 Review 和人工验收只登记为手工建议项，用户明确要求后再执行。

## 5. 拆分经验

遇到以下信号时，优先拆 Issue：

- 同时涉及前端、后端、数据库和 OpenAPI；
- 同时涉及权限、隐私、审核、日志和追踪；
- 同时出现状态机、异步任务、重试、暂停、取消、恢复点；
- 需要新增 Flyway 迁移并改变业务状态字段；
- 需要兼容旧接口、旧数据或旧前端入口；
- 需要引入临时 workflow 或专项 CI；
- 单个 PR 的验收标准超过 8 条，或任务看板超过 8 个原子任务。

推荐拆分顺序：

```text
领域模型 / 数据结构
→ API 契约
→ 后端最小闭环
→ 权限 / 隐私 / 审核治理
→ 前端页面接入
→ 追踪 / 日志 / 导出等增强
→ 手工验证 / 生产准出建议
```

## 6. 默认执行口径

后续执行 Issue 时默认遵循：

- 不主动扩大 VERIFY / REVIEW；
- 不主动触发高成本 E2E；
- 不把手工验证项当作已验证结果；
- 不把历史基线治理混入业务需求；
- 不把 UI 调整和后端治理闭环混成一个不可恢复的大任务；
- 如果发现需求进入重型区，先建议拆分或标记手工验证项，再继续当前最小任务。
