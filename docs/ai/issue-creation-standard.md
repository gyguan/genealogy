# Issue 创建、分组与执行顺序规范

本文档定义在 `gyguan/genealogy` 仓库中创建 Issue 的统一规则，重点解决同一功能点拆分多个 Issue 时命名不一致、顺序不清晰、依赖关系缺失和后续恢复困难的问题。

适用范围：

- 创建单个功能、缺陷、治理或文档 Issue；
- 将同一功能点拆分为多个 Issue；
- 基于聊天批量创建 Issue；
- 从分析报告、PR Review、CI 失败或架构治理中沉淀后续任务。

---

## 1. 核心原则

1. **先分组，再建单**：同一功能点拆分多个 Issue 前，必须先明确统一功能前缀、总目标和拆分边界。
2. **标题可排序**：同一功能点下的 Issue 标题必须包含统一前缀和执行序号。
3. **顺序可恢复**：每个 Issue 必须说明执行顺序、前置依赖、是否可并行和下一步 Issue。
4. **边界可验收**：每个 Issue 都必须有独立范围、非目标和验收标准。
5. **不得散落建单**：禁止同一功能点下出现多个互相关联但标题、描述、顺序完全无法识别的零散 Issue。
6. **复杂需求先切片**：涉及数据库、API、后端、前端、权限、审核、追踪、E2E 或 CI 的复杂需求，禁止直接塞进一个大 Issue，必须先按本文的复杂需求拆分规则切片。

---

## 2. Issue 标题规范

### 2.1 同一功能点的统一前缀

属于同一功能点的一组 Issue，标题统一使用：

```text
[<功能前缀> P<优先级>-<序号>] <动作 + 对象 + 结果>
```

示例：

```text
[来源资料库 P0-01] 建立来源资料分页搜索接口
[来源资料库 P0-02] 实现来源详情与引用关系查询
[来源资料库 P0-03] 接入前端列表、筛选和详情页
[来源资料库 P0-04] 补充权限、隐私和审计校验
```

字段说明：

| 字段 | 说明 | 示例 |
|---|---|---|
| 功能前缀 | 同一功能点的统一中文短名 | `来源资料库`、`成员权限`、`导入任务` |
| P 优先级 | 当前切片优先级，使用 P0/P1/P2 | `P0` |
| 序号 | 两位数字，表示建议执行顺序 | `01`、`02`、`03` |
| 标题正文 | 用动宾结构说明交付结果 | `建立来源资料分页搜索接口` |

### 2.2 单个独立 Issue

不属于某个功能组的独立 Issue，可以不使用功能组序号，但标题仍应明确模块和结果：

```text
[CI] 修复后端 PostgreSQL 启动检查失败
[文档] 补充数据库迁移命名规范
[权限] 修复成员列表跨支派可见风险
```

## 3. Issue 组创建规则

### 3.1 一组 Issue 必须先定义组信息

批量创建前，必须先确定：

- 功能前缀；
- 总目标；
- 范围和非目标；
- Issue 列表；
- 推荐执行顺序；
- 可并行项；
- 前置依赖；
- 完成后是否关闭总控 Issue 或进入下一阶段。

如果功能点超过 3 个 Issue，建议创建一个总控 Issue：

```text
[来源资料库 EPIC] 来源资料库搜索、维护、详情和引用关系重构
```

总控 Issue 用于记录完整任务清单、依赖和完成状态；具体实现仍拆到子 Issue 中。

### 3.2 Issue 正文必须包含执行顺序

同一功能组下的每个 Issue，正文必须包含：

```markdown
## 功能组

- 功能前缀：`来源资料库`
- 组内序号：`P0-02`
- 所属总控：#<EPIC_ISSUE_NUMBER> / 不适用
- 前置 Issue：#<ISSUE_NUMBER> / 无
- 后续 Issue：#<ISSUE_NUMBER> / 待创建 / 无
- 是否可并行：是 / 否
- 建议执行顺序：第 2 步，共 4 步
- 建议切片类型：领域模型 / API 契约 / 后端实现 / 前端接入 / 权限审核追踪 / E2E 准出 / 基线治理 / 文档治理
```

### 3.3 可并行必须明确说明

只有满足以下条件，才能标记为“可并行”：

- 不修改同一核心文件或同一 API 契约；
- 不依赖另一个 Issue 的数据库迁移或字段；
- 不依赖另一个 Issue 生成的 OpenAPI 类型；
- 不会导致同一页面、同一接口或同一领域规则反复冲突；
- 合入顺序不同不会破坏 `main`。

否则必须标记为“不可并行”，并写清楚前置 Issue。

### 3.4 复杂需求必须按标准层次拆分

当需求满足以下任一条件时，必须按标准层次拆分，不得直接创建单个大 Issue：

- 同时涉及数据库、API、后端、前端中的 3 类及以上；
- 涉及认证、成员权限、隐私、审核、审计、追踪、导入正式生效、生产准出或 CI 门禁；
- 需要新增 Flyway 迁移、OpenAPI 契约、生成类型、后端服务、前端页面和 E2E 中的多项；
- 需要同时实现新功能并治理历史基线问题；
- 预计需要完整后端测试、API Contract、前端构建、真实 PostgreSQL 或浏览器 E2E 才能完成验收。

标准拆分顺序如下：

| 顺序 | 切片类型 | 目标 | 典型验收 |
|---:|---|---|---|
| 1 | 领域模型 / 数据结构 | 明确对象、状态、兼容窗口、迁移或数据边界 | 文档、Flyway、实体、Repository 或兼容说明可独立验证 |
| 2 | API 契约 | 定义请求、响应、错误码、分页、权限和 OpenAPI 生成类型 | `api:generate` / `api:check` 通过，契约字段清晰 |
| 3 | 后端实现 | 完成应用服务、领域规则、Repository、权限前置校验和单元测试 | 后端编译、聚焦测试、必要的 PostgreSQL 测试通过 |
| 4 | 前端接入 | 完成页面、交互、状态恢复、错误展示和生成类型接入 | TypeScript、页面测试、前端构建通过 |
| 5 | 权限 / 隐私 / 审核 / 追踪接入 | 补齐越权防护、最小披露、审核流、审计日志和追踪链路 | 安全场景、审核状态、日志和追踪覆盖可验证 |
| 6 | E2E / 生产准出 | 建立真实环境、浏览器链路、CI 门禁、发布与回滚说明 | 真实 PostgreSQL、E2E、生产构建、准出文档通过 |

每个 Issue 原则上只承载 **一个主流程 + 一个验证闭环**。例如“后端导入草稿创建 + 后端聚焦测试”可以作为一个 Issue；“导入后端、前端、审核、权限、E2E 和历史 Flyway 修复”不能放在同一个 Issue。

### 3.5 基线治理必须与业务功能拆开

历史基线问题包括但不限于：

- 重复或冲突的 Flyway 迁移；
- JPA / entityManagerFactory / PostgreSQL 启动失败；
- 旧测试桩、旧权限码或旧 API 生成文件不一致；
- CI 触发范围过宽、E2E 过重或 workflow 结构失败；
- 历史页面 TypeScript 错误、构建基线错误或契约漂移。

这些问题如果不是当前功能的直接交付内容，必须单独创建 `[基线治理]`、`[CI]`、`[数据库]` 或 `[测试]` Issue，不能混入业务功能 Issue。业务 Issue 只能记录“受某基线 Issue 阻塞”或“依赖某基线 Issue 完成后继续”。

允许例外：如果当前业务功能本身的验收目标就是“恢复生产启动”“修复 Flyway 基线”“建设认证 E2E”，则可以作为该 Issue 的主目标，但必须在标题和范围中明确这是基线治理或准出治理任务。

## 4. Issue 正文模板

```markdown
## 背景

<!-- 为什么要做，当前问题是什么。 -->

## 功能组

- 功能前缀：`<功能前缀>`
- 组内序号：`P0-01`
- 所属总控：#<EPIC_ISSUE_NUMBER> / 不适用
- 前置 Issue：#<ISSUE_NUMBER> / 无
- 后续 Issue：#<ISSUE_NUMBER> / 待创建 / 无
- 是否可并行：是 / 否
- 建议执行顺序：第 1 步，共 N 步
- 建议切片类型：领域模型 / API 契约 / 后端实现 / 前端接入 / 权限审核追踪 / E2E 准出 / 基线治理 / 文档治理

## 目标

<!-- 本 Issue 完成后要交付什么。 -->

## 范围

- [ ] <!-- 范围 1 -->
- [ ] <!-- 范围 2 -->

## 非目标

- <!-- 明确本 Issue 不做什么，避免范围膨胀。 -->

## 验收标准

- [ ] <!-- 可验证标准 1 -->
- [ ] <!-- 可验证标准 2 -->

## 影响范围

- 后端：<!-- 模块 / 不涉及 -->
- 前端：<!-- 页面 / 不涉及 -->
- API：<!-- 是否涉及 OpenAPI -->
- 数据库：<!-- 是否涉及 Flyway -->
- 权限 / 隐私 / 审核：<!-- 是否涉及 -->
- CI / E2E：<!-- 是否涉及重型检查或生产准出 -->

## 风险与约束

- <!-- 权限、隐私、审核、兼容、性能、迁移等风险。 -->

## 验证建议

- <!-- 建议执行的测试、构建、契约或手工验证。 -->
```

## 5. 创建顺序

创建一组 Issue 时，必须按以下顺序执行：

1. 明确功能前缀和总目标；
2. 判断是否属于复杂需求；
3. 如果属于复杂需求，先按“领域模型 / API / 后端 / 前端 / 权限审核追踪 / E2E 准出”拆分；
4. 拆分 Issue 列表；
5. 排列执行顺序；
6. 标记前置依赖和可并行项；
7. 明确基线治理 Issue 是否需要单独创建；
8. 必要时创建总控 Issue；
9. 创建具体 Issue；
10. 回填总控 Issue 中的 Issue 清单；
11. 检查标题前缀、序号、切片类型和正文功能组信息是否一致。

禁止先随意创建多个 Issue，再事后补顺序和依赖。

---

## 6. 与实现规范的关系

- 本文档负责 **Issue 创建前的分组、命名、顺序和依赖**；
- `docs/ai/issue-execution-governance.md` 负责 **Issue 进入实现后的分支、Draft PR、任务看板和恢复**；
- `docs/ai/task-duration-standard.md` 负责 **执行任务看板、状态、耗时和恢复字段**。

创建 Issue 后，如果用户要求 `实现 Issue #N`，必须继续遵循 Issue 实现规范，不得因为 Issue 已经有顺序就跳过 Draft PR 和任务看板。

---

## 7. 禁止事项

禁止：

- 同一功能点下 Issue 标题没有统一前缀；
- 同一功能点下 Issue 没有执行序号；
- Issue 正文缺少前置依赖和是否可并行说明；
- 复杂需求缺少切片类型和标准拆分路径；
- 用“后端优化”“前端调整”“补充能力”等模糊标题替代可交付结果；
- 把强依赖任务标记为可并行；
- 把数据库迁移、API 契约、后端、前端、权限审核、E2E 和 CI 准出混在一个大 Issue 中；
- 把历史基线治理混入业务功能 Issue，除非该 Issue 的主目标就是基线治理；
- 创建多个 Issue 后不回填总控 Issue 或功能组关系；
- Issue 范围覆盖过大，导致无法独立验收。
