# Genealogy 通用 AI 提问模板库

本文档沉淀 `gyguan/genealogy` 项目后续研发、分析、Review、修复和设计时的通用提问模板。

使用目标：

- 让 AI 自动读取并遵守 `AGENTS.md`。
- 让 AI 按 `docs/ai/ai-engineering-workflow.md` 中的 `DEFINE → PLAN → BUILD → VERIFY → REVIEW` 流程工作。
- 让 AI 根据 `docs/ai/skill-mapping.md` 自动选择合适的工程 Skill。
- 让需求、代码修改、PR Review、问题修复和架构分析都能复用同一套仓库规范。

---

## 0. 通用启动语

适用于所有任务，建议每次提问开头都带上：

```text
请读取 GitHub 仓库 gyguan/genealogy 的 main 分支，优先遵守 AGENTS.md、docs/ai/ai-engineering-workflow.md、docs/ai/skill-mapping.md 中定义的 AI 工程流程和项目规范。

本次任务请先判断任务类型，选择适合的工程 Skill，按 DEFINE → PLAN → BUILD → VERIFY → REVIEW 流程处理，不要直接跳到编码。

如果需要修改代码，请新建分支实现，并提交 PR，不要直接修改 main。
```

## 5. 代码评审 / PR Review 模板

适用于：让 AI Review 某个 PR、某个分支、某段 diff。

```text
请读取 GitHub 仓库 gyguan/genealogy 的 PR：【填写 PR 编号】。

本次任务类型：代码评审 / PR Review。

请遵守 AGENTS.md、.github/pull_request_template.md 和 docs/ai/skill-mapping.md，按 code-review-and-quality 的五轴 Review：

1. Correctness：是否满足需求和验收标准；
2. Readability：命名、结构、控制流是否清晰；
3. Architecture：是否符合 Controller / Application Service / Domain Service / Repository 分层，以及前端 Ant Design 规范；
4. Security：是否有权限、隐私、注入、敏感信息泄露风险；
5. Performance：是否有 N+1、无分页列表、无边界递归、大对象渲染等风险。

重点关注：
【例如：是否破坏审核流程】
【例如：是否只在前端做权限控制】
【例如：是否修改 API 但未更新 OpenAPI】
【例如：是否暴露技术 ID】

请输出：
1. 总体结论：可合入 / 需修改 / 风险较高；
2. 必须修改项；
3. 建议优化项；
4. 可忽略的小问题；
5. 验证充分性；
6. 是否建议补测试或补文档。
```

## 16. 使用建议

1. 需求越模糊，越要使用“需求新增 / 功能新增模板”。
2. 已经知道要改哪个页面时，使用“前端页面 / 交互改造模板”。
3. 发现报错时，使用“问题修复 / Bug 修复模板”，并尽量提供日志和复现步骤。
4. 只想让 AI 分析仓库时，务必明确“不改代码、不建分支、不提交 PR”。
5. 涉及权限、隐私、审核、导入导出、附件、世系图谱时，一定要写清楚风险约束和验收标准。
