# Issue 实现、状态持久化与中断恢复规范

本文档定义通过聊天连接 GitHub 实现 Issue 时的强制执行顺序。目标是确保会话中断、页面卡死或更换会话后，任何人或 AI 都能仅依赖 GitHub 恢复当前工作现场，并在 Issue 实现完成后按门禁直接合入 `main`。

适用于：

- `实现 Issue #N`；
- `继续 Issue #N`；
- 与 Issue 关联的功能开发、缺陷修复、架构治理和 CI 修复；
- 无法使用本地开发环境、主要通过聊天和 GitHub 连接执行的任务。

执行任务看板、任务状态、活跃耗时和耗时汇总的唯一权威口径为：`docs/ai/task-duration-standard.md`。

Issue 类型、流程强度、契约强度、验证强度、拆分信号、活跃耗时 / 外部等待分离和轻量 / 重型流程取舍，必须同时遵循 `docs/ai/issue-delivery-cost-experience.md`。实现或继续 Issue 时，不得把轻量 UI 任务强行套成重型 Review，也不得把数据库、权限、隐私、审核、Contract First、Tree 核心链路或跨模块状态机降级为轻量流程；确需偏离时必须在任务看板和 PR 中说明原因、风险和补偿验证。

当本规范与其他 AI 协作文档存在冲突时，Issue 实现和恢复流程以本文档为准；但不得覆盖根 `AGENTS.md` 的 P0/P1 规则。

---

## 1. 状态载体与唯一职责

| 载体 | 唯一职责 |
|---|---|
| Issue 正文 | 原始需求、范围、约束和验收标准 |
| Issue 评论 | 启动、阻塞、重要里程碑和最终完成摘要 |
| 任务分支 | 隔离当前 Issue 的代码与文档变更 |
| `tasks/issue-<N>-execution.md` | 可版本化的执行计划、任务看板、耗时、验证和恢复检查点 |
| Draft PR 描述 | 当前对外可见的任务看板、耗时、验证结果、风险和恢复状态 |
| Commit | 已实际完成的原子变更 |
| CI | 构建、测试、契约和静态检查证据 |
| Review | 评审意见及整改状态 |
| `main` 合入 Commit | Issue 最终交付完成的事实依据 |

不得使用聊天记录作为唯一项目记忆。聊天中出现但未写入 GitHub 的“已完成”状态或耗时，不视为可恢复的正式进度。

## 2. 实现 Issue 的强制启动门禁

收到 `实现 Issue #N` 后，必须严格按以下顺序执行。Gate 1～7 未全部完成前，禁止修改业务代码。

### Gate 1：刷新最新规则

1. 从 `main` 重新读取最新 `AGENTS.md`；
2. 读取本文档和 `docs/ai/task-duration-standard.md`；
3. 读取 `docs/ai/issue-delivery-cost-experience.md`，确认本 Issue 的类型、流程强度、契约强度、验证强度和拆分信号；
4. 按任务范围读取目录级 `AGENTS.md` 和专项规范；
5. 不得继续沿用会话早期缓存的仓库规则；
6. 如果 `main` 在当前会话期间发生过合入，执行下一阶段前必须再次刷新相关规则。

### Gate 2：读取需求和已有现场

1. 读取 Issue 正文和全部评论；
2. 搜索关联的开放 PR、远程分支、Commit、CI 和 Review；
3. 判断当前属于首次启动、继续执行、缺少 PR、缺少看板、缺少耗时或缺少恢复点中的哪一种。

不得在未检查现有现场前重新创建重复分支、重复 PR 或重复实现。

### Gate 3：建立任务看板

根据 Issue 验收标准拆解原子任务，并建立：

```text
tasks/issue-<N>-execution.md
```

该文件至少包含：

- Issue 链接与目标；
- 本次实现范围和非目标；
- Issue 类型、流程强度、契约强度、验证强度；
- 是否命中 `docs/ai/issue-delivery-cost-experience.md` 的拆分信号，以及实际拆分 / 不拆分结论；
- 遵循 `docs/ai/task-duration-standard.md` 的任务看板；
- 活跃耗时与外部等待分离口径；
- 复用的测试 fixture、helper、模板或无法复用的原因；
- 影响模块；
- 验证方案；
- 已知风险；
- 当前恢复检查点；
- 最后更新时间，统一使用北京时间。

### Gate 4：创建或恢复远程分支

默认分支命名：

```text
agent/issue-<N>-<简短描述>
```

分支必须从最新 `main` 创建。发现同一 Issue 已有有效远程分支时，应优先恢复，不得另建平行分支。

### Gate 5：先提交执行检查点

在修改业务代码前，先提交 `tasks/issue-<N>-execution.md`。

该提交用于：

- 让任务分支相对 `main` 产生可创建 PR 的差异；
- 持久化首次任务看板和计时起点；
- 为会话中断提供最早恢复点。

禁止以“分支尚无代码”为理由推迟创建 Draft PR。

### Gate 6：立即创建 Draft PR

检查点提交后，必须立即创建 Draft PR。完成 Draft PR 之前，禁止修改业务代码。

Draft PR 必须包含：

- Issue 关联：`Refs #N`、`Closes #N`、`Fixes #N` 或 `Resolves #N`；
- 遵循 `docs/ai/task-duration-standard.md` 的当前任务看板；
- 遵循 `docs/ai/issue-delivery-cost-experience.md` 的 Issue 类型、流程强度、契约强度、验证强度和拆分结论；
- 当前进行中任务及其累计耗时；
- 验证方案和当前验证结果；
- 已知风险；
- 耗时汇总；
- 外部等待说明；
- 恢复检查点；
- 当前分支；
- 下一步最小任务。

默认一个 Issue 对应一个 Draft PR。确需拆分多个 PR 时：

- 每个切片 PR 使用 `Refs #N`；
- PR 描述明确切片范围和依赖顺序；
- 最终集成 PR 使用 `Closes #N`，或在全部切片合入后手工回写并关闭 Issue。

### Gate 7：回写 Issue

Draft PR 创建成功后，立即在 Issue 评论中写入：

- 已进入实现阶段；
- 工作分支；
- Draft PR 链接；
- 当前执行阶段；
- 详细任务看板位置；
- 详细进度、耗时与恢复检查点位置。

## 3. 标准启动评论

```markdown
已进入实现阶段。

- 工作分支：`agent/issue-<N>-<slug>`
- Draft PR：#<PR_NUMBER>
- 当前阶段：需求与影响范围确认
- 执行看板：`tasks/issue-<N>-execution.md`
- 详细进度、耗时与恢复检查点：见 Draft PR 描述
```

不得只写计划中的分支名。评论中的分支和 PR 必须已经在 GitHub 真实存在。

## 4. 原子任务执行闭环

每个原子任务必须按以下循环推进：

```text
读取相关代码与文档
  ↓
确认本任务输入、输出和验收条件
  ↓
开始或恢复活跃计时
  ↓
修改有限范围文件
  ↓
检查 diff，删除无关修改
  ↓
补充或更新测试
  ↓
执行可用验证
  ↓
形成独立 Commit
  ↓
停止计时并固化累计耗时
  ↓
更新任务文件和 Draft PR 看板
  ↓
再进入下一任务
```

每完成一个原子任务，至少更新：

- 任务状态；
- 实际修改文件或模块；
- 活跃耗时；
- 外部等待或未可验证耗时说明；
- Commit；
- 验证结果；
- 新发现的风险；
- 下一步最小任务；
- 北京时间的最后更新时间。

禁止连续完成多个大任务后，最后一次性补写进度。

---

## 5. Issue 回写时机

Issue 不承载每个文件级细节，只在以下里程碑回写评论：

### 开始实现

Draft PR 创建后立即回写。

### 出现阻塞

出现以下任一情况时立即回写：

- 需要用户进行关键产品或架构决策；
- 需要扩大 Issue 范围；
- 公共 API、数据库、权限、隐私或审核流程发生重大变化；
- CI 或环境问题导致任务无法继续；
- 原方案不可行，需要重拆任务。

### 完成重要阶段

例如后端接口完成、前端接入完成、迁移验证完成或进入最终 Review。

### 最终完成

Issue 实现 PR 合入 `main` 后，必须立即回写最终完成摘要：

- 合入 PR；
- `main` 合入 Commit；
- 完成内容；
- 验证结果；
- 活跃耗时汇总；
- 已知风险；
- 未完成或后续事项；
- Issue 是否已自动关闭；
- 北京时间的完成时间。

## 6. 会话中断与恢复

### 6.1 中断前检查点

每个原子任务提交后、进入下一任务前，必须更新恢复检查点。恢复检查点至少包括：

- 当前 Issue；
- 当前分支；
- 当前 Draft PR；
- 最后完成任务；
- 当前进行中任务；
- 当前任务累计耗时；
- 最新 Commit；
- CI 状态；
- 未解决 Review；
- 已知阻塞；
- 下一步最小任务；
- 最后更新时间（北京时间）。

无法预知页面突然中断，因此恢复能力依赖“每个原子任务完成即写回”，而不是等会话结束时再保存。

### 6.2 新会话恢复顺序

收到 `继续 Issue #N` 时：

1. 重新读取 `main/AGENTS.md` 和本文档；
2. 读取 Issue 正文与评论；
3. 查找关联开放 PR；
4. 读取 PR 描述、diff、Commit、CI 和 Review；
5. 读取 `tasks/issue-<N>-execution.md`；
6. 输出恢复检查点；
7. 从下一步最小任务继续。

不得重新完整分析仓库，也不得重复执行已完成任务。

### 6.3 历史任务补救

对于已经开始但未按规范执行的 Issue：

1. 暂停新的业务代码修改；
2. 检查已有分支、Commit 和 PR；
3. 根据实际代码反向建立 `tasks/issue-<N>-execution.md`；
4. 创建或补全 Draft PR；
5. 将 PR 链接和当前阶段回写 Issue；
6. 补齐任务看板、耗时、验证结果和恢复检查点；
7. 再继续后续实现。

不得因为任务已经进行到一半而跳过治理补齐。

## 7. PR 完成与自动合入门禁

Issue 实现 PR 满足合入条件后，默认必须直接合入 `main`，不再额外等待用户二次确认。

### 7.1 合入前必须满足

合入前必须同时满足：

- PR 已关联正确 Issue；
- 任务看板与实际 Commit、diff 一致；
- 本 PR 范围内没有未说明的“🔄 进行中”或“⚠️ 阻塞”；
- 活跃耗时与耗时汇总已更新；
- 恢复检查点已更新为最终状态；
- 验证结果真实记录，失败项已修复或明确为与本次无关的历史基线问题；
- Review 意见已处理，或明确记录为不阻塞合入的遗留事项；
- Issue 验收标准已逐项核对；
- 完整交付 PR 使用 `Closes/Fixes/Resolves #N`；
- 部分切片 PR 使用 `Refs #N`，不得误关闭 Issue。

### 7.2 默认合入动作

满足 7.1 后，AI 必须执行：

1. 将 Draft PR 转为 Ready；
2. 使用仓库默认合入方式，优先采用 Squash Merge；
3. 合入目标分支为 `main`；
4. 合入后读取 PR 状态，确认 `merged=true`；
5. 将合入 Commit、完成摘要、验证结果和北京时间回写 Issue。

该动作不等同于直接向 `main` 推送业务代码；所有业务变更仍必须通过 PR 合入。

### 7.3 禁止自动合入的情况

出现以下任一情况时，不得自动合入，必须在 PR 和 Issue 中说明阻塞原因：

- 用户在本次任务中明确要求“只创建 PR / 不合入 / 只分析”；
- PR 未满足 P0/P1 规则或 Issue 验收标准；
- CI、构建、测试、契约或安全检查存在未解释失败；
- 存在未处理的阻塞性 Review 意见；
- 存在未关闭的权限、隐私、审核、数据库或兼容性高风险；
- 仓库分支保护或 GitHub 权限阻止合入；
- 合入前发现 `main` 已更新且可能影响当前 PR，需先重新检查或同步。

### 7.4 合入后确认

合入后必须确认：

- PR 状态为 merged；
- 变更已进入 `main`；
- Issue 已自动关闭或手工更新状态；
- Issue 中存在最终完成摘要；
- 如未自动关闭 Issue，必须说明原因和后续处理方式。

## 8. 自动检查

仓库通过 `.github/workflows/issue-delivery-governance.yml` 对符合以下分支格式的 PR 执行结构检查：

```text
agent/issue-<N>-*
feature/issue-<N>-*
fix/issue-<N>-*
```

检查内容包括：

- PR 是否关联同一 Issue；
- 是否包含执行任务看板；
- 是否替换默认占位任务；
- 是否包含当前进行中、下一步任务和恢复检查点；
- 是否包含耗时列和耗时汇总；
- 是否包含验证结果；
- 是否保留未替换的模板占位内容。

该自动检查只能验证结构，不能替代人工核对任务状态、代码正确性、耗时真实性和验收结果。建议将该检查配置为 `main` 分支保护的必需状态检查。

---

## 9. 最小用户输入

规范生效后，用户只需输入：

```text
实现 Issue #104
```

AI 必须自动完成规则刷新、现场检查、任务看板、分支、Draft PR、Issue 回写、实现、验证、Review 自检、满足门禁后的 `main` 合入和最终完成回写，不再要求用户重复填写固定模板或再次确认合入。

如果用户输入：

```text
继续 Issue #101
```

AI 必须先恢复 GitHub 现场，再从下一步最小任务继续；若恢复后发现 Issue 已实现完成且 PR 满足合入门禁，应直接完成 `main` 合入和最终回写。
