# Issue 实现、状态持久化与中断恢复规范

本文档定义通过聊天连接 GitHub 实现 Issue 时的强制执行顺序。目标是确保会话中断、页面卡死或更换会话后，任何人或 AI 都能仅依赖 GitHub 恢复当前工作现场。

本规范适用于：

- `实现 Issue #N`；
- 继续尚未完成的 Issue；
- 与 Issue 关联的功能开发、缺陷修复、架构治理和 CI 修复；
- 无法使用本地开发环境、主要通过聊天和 GitHub 连接执行的任务。

当本规范与其他 AI 协作文档存在冲突时，Issue 实现和恢复流程以本文档为准。任务耗时的定义和展示细则以 `docs/ai/task-duration-standard.md` 为准。

---

## 1. 状态载体与唯一职责

| 载体 | 唯一职责 |
|---|---|
| Issue 正文 | 原始需求、范围、约束和验收标准 |
| Issue 评论 | 启动、阻塞、重要里程碑和最终完成摘要 |
| 任务分支 | 隔离当前 Issue 的代码与文档变更 |
| `tasks/issue-<N>-execution.md` | 可版本化的执行计划、任务看板、耗时和恢复检查点 |
| Draft PR 描述 | 当前对外可见的任务看板、耗时、验证结果、风险和恢复状态 |
| Commit | 已实际完成的原子变更 |
| CI | 构建、测试、契约和静态检查证据 |
| Review | 评审意见及整改状态 |

不得使用聊天记录作为唯一项目记忆。聊天中出现但未写入 GitHub 的“已完成”状态或耗时，不视为可恢复的正式进度。

---

## 2. 实现 Issue 的强制启动门禁

收到 `实现 Issue #N` 后，必须严格按以下顺序执行。

### Gate 1：刷新最新规则

1. 从 `main` 重新读取最新 `AGENTS.md`；
2. 读取本文档和 `docs/ai/task-duration-standard.md`；
3. 不得继续沿用会话早期缓存的仓库规则；
4. 如果 `main` 在当前会话期间发生过合入，执行下一阶段前必须再次刷新相关规则。

### Gate 2：读取需求和已有现场

1. 读取 Issue 正文和全部评论；
2. 搜索关联的开放 PR、远程分支、Commit、CI 和 Review；
3. 判断当前属于：
   - 首次启动；
   - 已有 Draft PR，继续执行；
   - 只有分支或 Commit、缺少 PR；
   - 只有 Issue 评论、没有远程执行现场；
   - 已有 PR 但任务看板、耗时或恢复点缺失或过期。

不得在未检查现有现场前重新创建重复分支、重复 PR 或重复实现。

### Gate 3：建立任务看板

根据 Issue 验收标准拆解原子任务，并建立：

```text
tasks/issue-<N>-execution.md
```

该文件至少包含：

- Issue 链接与目标；
- 本次实现范围和非目标；
- 包含“耗时”列的任务看板；
- 影响模块；
- 验证方案；
- 已知风险；
- 耗时口径或对 `docs/ai/task-duration-standard.md` 的引用；
- 当前恢复检查点；
- 最后更新时间，统一使用北京时间。

首次建立时：

- 待处理任务耗时填写 `—`；
- 当前进行中任务开始累计活跃耗时；
- 规范生效前无法还原的历史任务填写 `未记录（规范生效前）`。

### Gate 4：创建远程分支

默认分支命名：

```text
agent/issue-<N>-<简短描述>
```

分支必须从最新 `main` 创建。发现同一 Issue 已有有效远程分支时，应优先恢复，不得另建平行分支。

### Gate 5：先提交执行检查点

在修改业务代码前，先提交 `tasks/issue-<N>-execution.md`。

该提交的作用是：

- 让任务分支相对 `main` 产生可创建 PR 的差异；
- 持久化首次任务看板和计时起点；
- 为会话中断提供最早恢复点。

禁止以“分支尚无代码”为理由推迟创建 Draft PR。

### Gate 6：立即创建 Draft PR

检查点提交后，必须立即创建 Draft PR。完成 Draft PR 之前，禁止修改业务代码。

Draft PR 必须包含：

- Issue 关联：`Refs #N`、`Closes #N`、`Fixes #N` 或 `Resolves #N`；
- 包含“耗时”列的当前任务看板；
- 当前进行中任务及其累计耗时；
- 验证方案和当前验证结果；
- 已知风险；
- 耗时汇总；
- 恢复检查点；
- 当前分支；
- 下一步最小任务。

默认一个 Issue 对应一个 Draft PR。确需拆分多个 PR 时：

- 每个切片 PR 使用 `Refs #N`；
- PR 描述明确切片范围和依赖顺序；
- 最终集成 PR 使用 `Closes #N`，或在全部切片合入后手工回写并关闭 Issue。

### Gate 7：回写 Issue

Draft PR 创建成功后，立即在 Issue 评论中写入：

- 已进入实现阶段；
- 工作分支；
- Draft PR 链接；
- 当前执行阶段；
- 详细任务看板位置。

只有 Gate 1～7 全部完成，才允许进入业务代码修改。

## 3. 标准启动评论

```markdown
已进入实现阶段。

- 工作分支：`agent/issue-<N>-<slug>`
- Draft PR：#<PR_NUMBER>
- 当前阶段：需求与影响范围确认
- 执行看板：`tasks/issue-<N>-execution.md`
- 详细进度、耗时与恢复检查点：见 Draft PR 描述
```

不得只写计划中的分支名。评论中的分支和 PR 必须已经在 GitHub 真实存在。

---

## 4. 执行看板规范

### 4.1 标准格式

```markdown
| 序号 | 任务 | 状态 | 耗时 | Commit / 结果或说明 |
|---|---|---|---|---|
| 1 | 读取规则与需求 | ✅ 已完成 | 约 4 分钟 | 已确认范围 |
| 2 | 实现分页查询 | 🔄 进行中 | 已累计约 12 分钟 | 正在补测试 |
| 3 | 执行完整验证 | ⏳ 待处理 | — |  |
```

### 4.2 状态与耗时

| 标识 | 状态 | 完成判定 | 耗时填写 |
|---|---|---|---|
| ✅ | 已完成 | 结果已提交并完成必要验证 | 累计实际活跃耗时 |
| 🔄 | 进行中 | 当前执行中，或已修改但未验证 | `已累计约 N 分钟` |
| ⏳ | 待处理 | 尚未开始 | `—` |
| ⚠️ | 阻塞 | 因权限、依赖、CI、信息或技术问题无法继续 | 保留阻塞前累计耗时 |
| ❌ | 失败 | 执行失败且尚未修复 | 记录失败前累计耗时 |
| ⏭️ | 已跳过 | 确认不需要执行，并说明原因 | 未执行填 `—`；已分析则记录分析耗时 |

代码已修改但未提交、未检查 diff 或未完成必要验证时，不得标记为“✅ 已完成”。

任务耗时只统计实际活跃执行时间。等待用户、CI 排队与自动运行、外部系统等待、会话中断和明确暂停不计入耗时，应在说明中单独记录。

### 4.3 最小任务粒度

任务必须描述可验证结果，例如：

- 下线旧全量用户目录并补隐私回归测试；
- 成员列表按操作者支派范围执行数据库分页；
- 将历史 `branch` 授权迁移为明确语义；
- 增加导入类型与文件格式独立字段；
- 完成 OpenAPI、前端筛选和兼容读取同步。

禁止只写“修改后端”“完善前端”“补测试”等无法判断完成条件的任务。

### 4.4 双写规则

详细执行状态和耗时同时维护在：

1. `tasks/issue-<N>-execution.md`；
2. Draft PR 描述。

二者发生不一致时：

- 以 GitHub 中最新 Commit、diff 和 CI 为事实依据；
- 耗时以任务文件中已记录的活跃区间和累计值为准；
- 立即修正任务文件和 PR 描述；
- 不得仅依据聊天记忆判定状态或补造耗时。

## 5. 原子任务执行闭环

每个原子任务必须按以下循环推进：

```text
读取相关代码与文档
  ↓
确认本任务输入、输出和验收条件
  ↓
开始或恢复活跃计时
  ↓
修改有限范围文件
  ↓
检查 diff，删除无关修改
  ↓
补充或更新测试
  ↓
执行可用验证
  ↓
形成独立 Commit
  ↓
停止计时并固化累计耗时
  ↓
更新任务文件和 Draft PR 看板
  ↓
再进入下一任务
```

每完成一个原子任务，至少更新：

- 任务状态；
- 实际活跃耗时；
- 显著外部等待；
- 实际修改文件或模块；
- Commit；
- 验证结果；
- 新发现的风险；
- 下一步最小任务；
- 北京时间的最后更新时间。

禁止连续完成多个大任务后，最后一次性补写状态或猜测耗时。

## 6. Issue 回写时机

Issue 不承载每个文件级细节，只在以下里程碑回写评论：

### 开始实现

Draft PR 创建后立即回写。

### 出现阻塞

出现以下任一情况时立即回写：

- 需要用户进行关键产品或架构决策；
- 需要扩大 Issue 范围；
- 公共 API、数据库、权限、隐私或审核流程发生重大变化；
- CI 或环境问题导致任务无法继续；
- 原方案不可行，需要重拆任务。

阻塞评论可以记录已累计活跃耗时和外部等待，但不得给出未经依据的剩余时间承诺。

### 完成重要阶段

例如后端接口完成、前端接入完成、迁移验证完成或进入最终 Review。阶段评论可附当前耗时汇总。

### 最终完成

PR 合入后回写：

- 合入 PR；
- 完成内容；
- 验证结果；
- 活跃耗时汇总和显著外部等待；
- 已知风险；
- 未完成或后续事项；
- `main` 合入 Commit；
- 北京时间的完成时间。

## 7. 会话中断与恢复

### 7.1 中断前检查点

每个原子任务提交后、进入下一任务前，必须更新恢复检查点。恢复检查点至少包括：

- 当前 Issue；
- 当前分支；
- 当前 Draft PR；
- 最后完成任务；
- 当前进行中任务；
- 当前任务累计活跃耗时；
- 最新 Commit；
- CI 状态；
- 未解决 Review；
- 已知阻塞；
- 下一步最小任务；
- 最后更新时间（北京时间）。

会话中断前应暂停当前活跃计时。无法预知页面突然中断时，恢复后不得把中断期间计入耗时。

### 7.2 新会话恢复顺序

收到 `继续 Issue #N` 时：

1. 重新读取 `main/AGENTS.md`、本文档和耗时规范；
2. 读取 Issue 正文与评论；
3. 查找关联开放 PR；
4. 读取 PR 描述、diff、Commit、CI 和 Review；
5. 读取 `tasks/issue-<N>-execution.md`；
6. 恢复最后累计耗时并输出恢复检查点；
7. 从下一步最小任务继续新的活跃计时区间。

不得重新完整分析仓库，不得重复执行已完成任务，也不得把会话中断时间补入任务耗时。

### 7.3 历史任务补救

对于已经开始但未按规范执行的 Issue：

1. 暂停新的业务代码修改；
2. 检查已有分支、Commit 和 PR；
3. 根据实际代码反向建立 `tasks/issue-<N>-execution.md`；
4. 创建或补全 Draft PR；
5. 将 PR 链接和当前阶段回写 Issue；
6. 补齐任务看板、验证结果和恢复检查点；
7. 无法可靠还原的历史耗时统一填写 `未记录（规范生效前）`；
8. 从下一次恢复或任务切换开始记录耗时；
9. 再继续后续实现。

不得因为任务已经进行到一半而跳过治理补齐，也不得根据 Commit 时间差猜测活跃耗时。

## 8. PR 合入门禁

Issue 实现 PR 转为 Ready 或合入前，必须满足：

- PR 已关联正确 Issue；
- 任务看板与实际 Commit、diff 一致；
- 每条任务均有符合规范的耗时值；
- 耗时汇总与任务明细一致；
- 本 PR 范围内没有未说明的“🔄 进行中”或“⚠️ 阻塞”；
- 恢复检查点已更新为最终状态；
- 验证结果真实记录，失败项已明确说明；
- Review 意见已处理或形成明确遗留项；
- Issue 验收标准已逐项核对；
- 完整交付 PR 使用 `Closes/Fixes/Resolves #N`；
- 部分切片 PR 使用 `Refs #N`，不得误关闭 Issue。

合入后必须确认：

- PR 状态为 merged；
- 变更已进入 `main`；
- Issue 已自动关闭或手工更新状态；
- Issue 中存在最终完成摘要。

## 9. 自动检查

仓库通过 `.github/workflows/issue-delivery-governance.yml` 对符合以下分支格式的 PR 执行结构检查：

```text
agent/issue-<N>-*
feature/issue-<N>-*
fix/issue-<N>-*
```

检查内容包括：

- PR 是否关联同一 Issue；
- 是否包含执行任务看板；
- 新规范生效后的 PR 是否包含“耗时”列；
- 任务行是否填写耗时；
- 是否替换默认占位任务；
- 是否包含当前进行中和恢复检查点；
- 是否填写当前任务累计耗时；
- 是否包含验证结果；
- 是否保留未替换的模板占位内容。

本规范生效前已经创建的 PR 不强制追溯历史耗时，但后续任务应从恢复时开始记录。

该自动检查只能验证结构，不能替代人工核对任务状态、耗时真实性、代码正确性和验收结果。建议将该检查配置为 `main` 分支保护的必需状态检查。

---

## 10. 最小用户输入

规范生效后，用户只需输入：

```text
实现 Issue #104
```

或：

```text
继续 Issue #101
```

AI 必须自动完成规则刷新、现场检查、任务看板、耗时记录、分支、Draft PR、Issue 回写和恢复，不再要求用户重复填写固定模板。
