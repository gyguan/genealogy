# 聊天式 GitHub 开发指南

本文档定义通过聊天连接 GitHub 完成分析、实现、恢复、CI 修复和 PR 收尾时的协作方式。

执行任务看板、状态、耗时和耗时汇总的唯一权威口径为：`docs/ai/task-duration-standard.md`。本文档不重复定义看板格式。

---

## 1. 核心原则

### 1.1 GitHub 是项目记忆，聊天不是

项目状态必须持续沉淀到 GitHub：

| 载体 | 承载内容 |
|---|---|
| `AGENTS.md` | 长期有效的架构、编码、测试和安全规则 |
| Issue | 需求、问题、范围和验收标准 |
| `tasks/` | 任务看板、活跃耗时、验证和恢复检查点 |
| Draft PR | 对外可见的进度、风险、验证和恢复状态 |
| Commit | 可回溯的原子变更 |
| GitHub Actions | 构建、测试和静态检查结果 |
| Review | 人工或 AI 的评审意见及整改记录 |
| 聊天 | 本轮指令、分析、决策和执行入口 |

页面中断或新开聊天后，应以 GitHub 中的 Issue、PR、任务文件、Commit、CI 和 Review 状态恢复，不依赖聊天记录猜测进度。

### 1.2 固定信息预置，用户只描述差异

仓库、默认分支、技术栈、目录结构、验证命令、权限边界和业务不变量，统一从项目规则中读取，不要求用户每次重复填写。

用户日常只需说明：

- 要分析什么；
- 要实现什么；
- 要继续哪个 Issue 或 PR；
- 本次特别关注什么。

### 1.3 小批量、可验证、可恢复

一次只推进一个原子任务或一个可验证的垂直切片。每个任务必须具备明确输出、验收标准、验证方式和恢复点。

---

## 2. 默认工作流

```text
一句话需求
  ↓
读取 AGENTS.md、目录级规则与必要专项文档
  ↓
读取 Issue / PR / tasks / CI / Review 现场
  ↓
建立或恢复执行看板
  ↓
DEFINE：确认目标、范围、成功标准和风险
  ↓
PLAN：拆分可独立验证的原子任务
  ↓
BUILD：一次实现一个任务
  ↓
VERIFY：执行测试、构建、契约和静态检查
  ↓
REVIEW：检查正确性、架构、安全、性能和可维护性
  ↓
更新 Draft PR、任务文件和最终看板
```

默认规则：

- “分析”类任务只分析，不修改代码；
- “实现、修改、修复”类任务使用独立分支和 Draft PR；
- “继续”类任务从现有 PR、任务文件、CI 和 Review 恢复；
- 能从仓库推断的信息不再重复询问用户；
- 涉及数据库、公共 API、权限、隐私、审核流程或兼容性变化时，必须明确说明影响；
- 不直接修改 `main`。

## 3. 常用短指令

| 指令 | 默认行为 |
|---|---|
| `分析：<一句业务问题>` | 读取相关规则、文档和代码，输出当前实现、问题、风险、建议和任务拆解，不修改仓库 |
| `实现：<一句需求>` | 定位影响范围，建立看板，创建分支和 Draft PR，按原子任务修改并验证 |
| `实现 Issue #N` | 按 `docs/ai/issue-execution-governance.md` 执行七道启动门禁 |
| `继续 PR #N` | 读取 PR、关联 Issue、任务文件、diff、Commit、CI 和 Review，先输出恢复检查点 |
| `修复 PR #N 的 CI` | 聚焦失败检查、失败日志、本批提交和直接相关代码 |
| `检视 PR #N，重点关注……` | 按 Correctness、Readability、Architecture、Security、Performance 输出结论 |
| `收尾 PR #N` | 对照 Issue 验收标准检查看板、diff、测试、CI、文档、风险和遗留项 |
| `汇报进度` | 输出完整执行看板和耗时汇总 |
| `只看未完成` | 只展示进行中、阻塞、失败和待处理任务 |

## 4. 执行看板要求

执行看板必须遵循 `docs/ai/task-duration-standard.md`，包括：

- 标准表格格式；
- 状态定义；
- 活跃耗时口径；
- 耗时汇总；
- 过渡规则；
- 禁止补造耗时。

任务应根据实际需求动态生成，不机械套用固定清单。动态新增任务时，不得静默扩大范围；必须说明新增原因、影响范围、风险和验证方式。

不再需要的任务标记为“⏭️ 已跳过”，不要从看板中删除。

## 5. 过程反馈

### 5.1 反馈模式

| 模式 | 反馈策略 |
|---|---|
| 安静模式 | 仅展示首次看板、阻塞信息和最终看板 |
| 标准模式 | 默认；每完成 2～3 个关键任务更新一次 |
| 密集模式 | 每完成一个关键任务更新一次 |
| 暂停反馈 | 继续执行，但不再发送过程看板 |

暂停聊天反馈不等于暂停 GitHub 中的状态和耗时持久化。

### 5.2 必须立即反馈的情况

- CI、构建、测试或契约检查失败；
- 实际实现与需求或文档不一致；
- 缺少权限或必要信息；
- 需要扩大修改范围；
- 需要调整数据库、公共 API、权限、隐私或审核流程；
- 工具、连接或页面异常；
- 需要用户作出关键产品或架构决策。

不得发送虚假进度、虚构剩余时间，或用未验证的百分比表示完成度。

## 6. Draft PR 作为持久化工作区

长任务应尽早建立 Draft PR，而不是等所有代码完成后再创建。

PR 描述至少持续维护：

```markdown
## 目标

## 执行任务看板

## 当前进行中

## 下一步最小任务

## 未完成项

## 耗时汇总

## 验证结果

## 已知风险

## 恢复检查点
```

每完成一个原子任务：

1. 检查本批 diff；
2. 删除无关修改；
3. 补充或更新测试；
4. 形成独立提交；
5. 查看 CI；
6. 更新任务文件、PR 看板、耗时和恢复检查点。

---

## 7. 页面中断与新会话恢复

恢复任务时，按以下顺序读取：

1. `AGENTS.md`；
2. 关联 Issue；
3. Draft PR 描述；
4. `tasks/` 中的执行文件；
5. 当前 diff；
6. Commit 记录；
7. CI 状态和失败信息；
8. Review 意见；
9. 上一次任务看板或恢复检查点。

恢复后先输出恢复检查点，再继续下一步最小任务。禁止重复执行已经完成的修改，以 GitHub 实际状态为准。

---

## 8. 适用边界

适合：

- 仓库分析；
- 功能开发与页面改造；
- Bug 和 CI 修复；
- PR Review；
- 文档、接口和测试同步；
- 无法使用本地开发环境的远程协作。

以下操作仍必须谨慎处理，并在执行前明确影响：

- 数据库 schema 或迁移变更；
- 公共 API 破坏性调整；
- 权限、隐私、审核流程修改；
- 删除历史兼容逻辑；
- 合并 PR、删除分支和关闭关键 Issue；
- 涉及真实个人隐私或敏感数据的操作。

---

## 9. 推荐落地组合

```text
AGENTS.md
  定义长期规则
      ↓
Issue
  定义需求和验收
      ↓
tasks/issue-<N>-execution.md
  保存任务看板、耗时和恢复检查点
      ↓
Draft PR
  对外展示进度、风险和验证结果
      ↓
原子 Commit
  保存每个小批量变更
      ↓
GitHub Actions
  提供可重复验证
      ↓
Review / Merge
  完成人工把关和交付
```

这套模式的核心不是让聊天一次完成所有工作，而是让每次聊天都能从 GitHub 恢复、推进一个明确任务，并把结果重新写回 GitHub。
