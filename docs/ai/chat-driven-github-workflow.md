# 聊天式 GitHub 开发指南

本文档定义通过聊天连接 GitHub 完成分析、实现、恢复、CI 修复和 PR 收尾时的协作方式。

执行任务看板、状态、耗时和耗时汇总的唯一权威口径为：`docs/ai/task-duration-standard.md`。本文档不重复定义看板格式。

---

## 1. 核心原则

### 1.1 GitHub 是项目记忆，聊天不是

项目状态必须持续沉淀到 GitHub：

| 载体 | 承载内容 |
|---|---|
| `AGENTS.md` | 长期有效的架构、编码、测试和安全规则 |
| Issue | 需求、问题、范围和验收标准 |
| `tasks/` | 任务看板、活跃耗时、验证和恢复检查点 |
| Draft PR | 对外可见的进度、风险、验证和恢复状态 |
| Commit | 可回溯的原子变更 |
| GitHub Actions | 构建、测试和静态检查结果 |
| Review | 人工或 AI 的评审意见及整改记录 |
| 聊天 | 本轮指令、分析、决策和执行入口 |

页面中断或新开聊天后，应以 GitHub 中的 Issue、PR、任务文件、Commit、CI 和 Review 状态恢复，不依赖聊天记录猜测进度。

### 1.2 固定信息预置，用户只描述差异

仓库、默认分支、技术栈、目录结构、验证命令、权限边界和业务不变量，统一从项目规则中读取，不要求用户每次重复填写。

用户日常只需说明：

- 要分析什么；
- 要实现什么；
- 要继续哪个 Issue 或 PR；
- 本次特别关注什么。

### 1.3 小批量、可验证、可恢复

一次只推进一个原子任务或一个可验证的垂直切片。每个任务必须具备明确输出、验收标准、验证方式和恢复点。

### 1.4 默认采用 ChatGPT 页面直连 GitHub 模式

当前协作模式是用户通过 ChatGPT 页面直接连接 GitHub，没有本地开发环境作为执行前提。

后续所有分析、实现、继续、CI 修复和收尾操作，默认必须基于 GitHub 远程状态推进：

- 读取仓库文件、Issue、PR、Commit、CI、Review 和任务文件；
- 通过 GitHub 提交文件变更、创建或更新 Issue / PR；
- 以 GitHub Actions、PR 状态、Review 结果和仓库文件内容作为验证依据；
- 不要求用户提供本地目录、本地命令输出或本地运行环境；
- 不尝试连接本地数据库、本地服务、本地 Git 仓库或本地开发服务器；
- 不把“请在本地运行”作为默认交付前提。

如果确实存在只能本地完成的验证，应明确标记为“本地不可执行，仅给出建议命令”，不得把它当作已完成验证。

## 2. 默认工作流

```text
一句话需求
  ↓
读取 AGENTS.md、目录级规则与必要专项文档
  ↓
读取 Issue / PR / tasks / CI / Review 现场
  ↓
建立或恢复执行看板
  ↓
DEFINE：确认目标、范围、成功标准和风险
  ↓
PLAN：拆分可独立验证的原子任务
  ↓
BUILD：一次实现一个任务
  ↓
VERIFY：按影响范围执行必要最小验证
  ↓
REVIEW：执行变更级快速复核
  ↓
更新 Draft PR、任务文件和最终看板
```

默认规则：

- “分析”类任务只分析，不修改代码；
- “拆分需求”类任务只创建或更新 EPIC / 子 Issue、顺序、依赖和验收标准，不实现代码；
- “创建单个 Issue”类任务只为一个明确问题创建一张独立 Issue，不拆多个 Issue，不实现代码；
- “实现、修改、修复”类任务使用独立分支和 Draft PR；
- “继续”类任务从现有 PR、任务文件、CI 和 Review 恢复；
- 所有操作默认基于 GitHub 远程状态，不依赖本地环境；
- 能从仓库推断的信息不再重复询问用户；
- 涉及数据库、公共 API、权限、隐私、审核流程或兼容性变化时，必须明确说明影响；
- 不直接修改 `main`。

### 2.1 需求执行路径

需求从提出到交付，默认按以下路径推进：

```text
分析需求 → 拆分需求为 Issue / 创建单个 Issue → 实现 Issue → 继续 Issue / PR → 修复 CI / 处理 Review → 收尾 PR
```

各路径默认边界如下：

| 路径 | 默认行为 | 主要产物 |
|---|---|---|
| 分析需求 | 读取规则、文档、代码和现有实现，只输出结论、风险、建议和是否建议编码 | 分析结论、风险等级、任务拆解 |
| 拆分需求为 Issue | 面向较大需求或复杂需求，按 `docs/ai/issue-creation-standard.md` 建立 EPIC / 子 Issue、执行顺序、依赖和验收标准 | EPIC、子 Issue 清单、依赖关系、验收标准 |
| 创建单个 Issue | 面向单一明确问题，只创建一张独立 Issue，写清目标、范围、非目标、影响范围、风险和验收标准 | 单个 Issue |
| 实现 Issue | 按 `docs/ai/issue-execution-governance.md` 建立任务文件、分支、Draft PR，并按原子任务实现 | `tasks/issue-N-execution.md`、Draft PR、Commit |
| 继续 Issue / PR | 从 GitHub 现场恢复，先输出恢复检查点，再推进下一步最小任务 | 恢复检查点、更新后的任务看板 |
| 修复 CI / 处理 Review | 只处理本 Issue 或 PR 直接相关的失败和 Review，历史基线问题单独记录或建单 | 修复提交、验证记录、阻塞说明 |
| 收尾 PR | 核对验收标准、diff、测试、CI、Review、风险和遗留项，满足门禁后收口 | 完成摘要、合入记录、Issue 回写 |

不得把分析、拆分、实现和继续混成一个不可恢复的大任务；如果用户只要求“分析”“拆分需求”或“创建单个 Issue”，不得提前进入代码实现。

### 2.2 轻量 VERIFY / REVIEW 规则

VERIFY 和 REVIEW 默认采用最小化策略，避免每个任务都进入全量测试和五轴评审。

| 环节 | 默认动作 | 手工执行条件 |
|---|---|---|
| VERIFY | 只检查本次变更直接相关的文件、契约或 GitHub Actions 状态；优先采用现有自动门禁结果 | 涉及数据库迁移、公共 API、权限、隐私、审核流程、认证、生产准出或跨模块重构时，只登记为手工验证项；专项测试、契约检查、构建或 E2E 由用户明确要求后再手工触发 |
| REVIEW | 只做变更级快速复核：是否满足 Issue、是否有明显副作用、是否触碰红线 | 涉及高风险领域或大范围改造时，只登记为手工 Review 项；Correctness / Architecture / Security / Performance / Maintainability 五轴 Review 由用户明确要求后再手工执行 |

轻量变更的默认验证口径：

- 文案、菜单、样式和文档变更：检查 diff、影响范围和现有自动门禁状态；
- 前端页面变更：以前端构建门禁和相关页面代码复核为主；
- 后端非 API 变更：以后端编译打包门禁和相关代码复核为主；
- API 契约变更：只依赖 API Contract 自动门禁；
- Flyway 迁移变更：只依赖 Database Migration Governance 自动门禁；
- CI / workflow 变更：检查触发条件、路径过滤、并发取消和是否符合最小门禁目标。

禁止把“执行测试、构建、契约和静态检查”理解为每次都全量运行；禁止把“检查正确性、架构、安全、性能和可维护性”理解为每次都输出完整五轴评审。高风险场景默认只记录手工验证 / 手工 Review 建议，不自动执行专项测试、E2E 或完整五轴 Review。

## 3. 常用短指令

| 指令 | 默认行为 |
|---|---|
| `分析：<一句业务问题>` | 读取相关规则、文档和代码，输出当前实现、问题、风险、建议和任务拆解，不修改仓库 |
| `拆分需求：<较大需求>` | 将较大或复杂需求拆成 EPIC / 子 Issue，维护执行顺序、前后依赖和验收标准，不实现代码 |
| `创建单个 Issue：<明确问题>` | 为一个明确问题创建一张独立 Issue，写清目标、范围、非目标、影响范围、风险和验收标准，不实现代码 |
| `实现：<一句需求>` | 定位影响范围，建立看板，创建分支和 Draft PR，按原子任务修改并验证 |
| `实现 Issue #N` | 按 `docs/ai/issue-execution-governance.md` 执行七道启动门禁 |
| `继续 Issue #N` | 从 Issue、已有 PR、任务文件、CI 和 Review 恢复，先输出恢复检查点再推进 |
| `继续 PR #N` | 读取 PR、关联 Issue、任务文件、diff、Commit、CI 和 Review，先输出恢复检查点 |
| `修复 PR #N 的 CI` | 聚焦失败检查、失败日志、本批提交和直接相关代码 |
| `检视 PR #N，重点关注……` | 按 Correctness、Readability、Architecture、Security、Performance 输出结论 |
| `收尾 PR #N` | 对照 Issue 验收标准检查看板、diff、测试、CI、文档、风险和遗留项 |
| `汇报进度` | 输出完整执行看板和耗时汇总 |
| `只看未完成` | 只展示进行中、阻塞、失败和待处理任务 |

## 4. 执行看板要求

执行看板必须遵循 `docs/ai/task-duration-standard.md`，包括：

- 标准表格格式；
- 状态定义；
- 活跃耗时口径；
- 耗时汇总；
- 过渡规则；
- 禁止补造耗时。

任务应根据实际需求动态生成，不机械套用固定清单。动态新增任务时，不得静默扩大范围；必须说明新增原因、影响范围、风险和验证方式。

不再需要的任务标记为“⏭️ 已跳过”，不要从看板中删除。

## 5. 过程反馈

### 5.1 反馈模式

| 模式 | 反馈策略 |
|---|---|
| 安静模式 | 仅展示首次看板、阻塞信息和最终看板 |
| 标准模式 | 默认；每完成 2～3 个关键任务更新一次 |
| 密集模式 | 每完成一个关键任务更新一次 |
| 暂停反馈 | 继续执行，但不再发送过程看板 |

暂停聊天反馈不等于暂停 GitHub 中的状态和耗时持久化。

### 5.2 必须立即反馈的情况

- CI、构建、测试或契约检查失败；
- 实际实现与需求或文档不一致；
- 缺少权限或必要信息；
- 需要扩大修改范围；
- 需要调整数据库、公共 API、权限、隐私或审核流程；
- 工具、连接或页面异常；
- 需要用户作出关键产品或架构决策。

不得发送虚假进度、虚构剩余时间，或用未验证的百分比表示完成度。

## 6. Draft PR 作为持久化工作区

长任务应尽早建立 Draft PR，而不是等所有代码完成后再创建。

PR 描述至少持续维护：

```markdown
## 目标

## 执行任务看板

## 当前进行中

## 下一步最小任务

## 未完成项

## 耗时汇总

## 验证结果

## 已知风险

## 恢复检查点
```

每完成一个原子任务：

1. 检查本批 diff；
2. 删除无关修改；
3. 补充或更新测试；
4. 形成独立提交；
5. 查看 CI；
6. 更新任务文件、PR 看板、耗时和恢复检查点。

## 7. 页面中断与新会话恢复

恢复任务时，按以下顺序读取：

1. `AGENTS.md`；
2. 关联 Issue；
3. Draft PR 描述；
4. `tasks/` 中的执行文件；
5. 当前 diff；
6. Commit 记录；
7. CI 状态和失败信息；
8. Review 意见；
9. 上一次任务看板或恢复检查点。

恢复后先输出恢复检查点，再继续下一步最小任务。禁止重复执行已经完成的修改，以 GitHub 实际状态为准。

---

## 8. 适用边界

适合：

- 仓库分析；
- 需求拆分与 Issue 创建；
- 功能开发与页面改造；
- Bug 和 CI 修复；
- PR Review；
- 文档、接口和测试同步；
- 无法使用本地开发环境的远程协作。

以下操作仍必须谨慎处理，并在执行前明确影响：

- 数据库 schema 或迁移变更；
- 公共 API 破坏性调整；
- 权限、隐私、审核流程修改；
- 删除历史兼容逻辑；
- 合并 PR、删除分支和关闭关键 Issue；
- 涉及真实个人隐私或敏感数据的操作。

## 9. 推荐落地组合

```text
AGENTS.md
  定义长期规则
      ↓
Issue
  定义需求和验收
      ↓
tasks/issue-<N>-execution.md
  保存任务看板、耗时和恢复检查点
      ↓
Draft PR
  对外展示进度、风险和验证结果
      ↓
原子 Commit
  保存每个小批量变更
      ↓
GitHub Actions
  提供可重复验证
      ↓
Review / Merge
  完成人工把关和交付
```

这套模式的核心不是让聊天一次完成所有工作，而是让每次聊天都能从 GitHub 恢复、推进一个明确任务，并把结果重新写回 GitHub。
