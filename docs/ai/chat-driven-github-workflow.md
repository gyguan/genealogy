# 聊天式 GitHub 开发与任务看板使用指南

本文档用于规范在**无法将代码下载到本地、主要通过聊天连接 GitHub 完成分析和编码**时的协作方式。

目标是让使用者只需输入一句业务诉求，AI 自动完成仓库定位、规则读取、任务拆解、过程反馈、代码修改、验证和 PR 交付，同时避免长任务卡死、上下文丢失和重复执行。

任务耗时的定义、计时和展示细则以 `docs/ai/task-duration-standard.md` 为准。

---

## 1. 核心原则

### 1.1 GitHub 是项目记忆，聊天不是

项目状态必须持续沉淀到 GitHub：

| 载体 | 承载内容 |
|---|---|
| `AGENTS.md` | 长期有效的架构、编码、测试和安全规则 |
| Issue | 需求、问题、范围和验收标准 |
| Draft PR | 当前任务看板、耗时、执行进度、风险和待办 |
| Commit | 可回溯的原子变更 |
| GitHub Actions | 构建、测试和静态检查结果 |
| Review | 人工或 AI 的评审意见及整改记录 |
| 聊天 | 本轮指令、分析、决策和执行入口 |

页面中断或新开聊天后，应以 GitHub 中的 Issue、PR、Commit、CI 和 Review 状态恢复，不依赖聊天记录猜测进度或耗时。

### 1.2 固定信息预置，用户只描述差异

仓库、默认分支、技术栈、目录结构、验证命令、权限边界和业务不变量，统一从项目规则中读取，不要求用户每次重复填写。

用户日常只需说明：

- 要分析什么；
- 要实现什么；
- 要继续哪个 Issue 或 PR；
- 本次特别关注什么。

### 1.3 阶段驱动，时间反馈为辅

长任务应按关键阶段持续反馈，不承诺无法保证的精确时间间隔，不使用没有依据的完成百分比，也不提供未经发生的预计耗时。

默认节奏：

1. 任务开始时建立带“耗时”列的任务看板；
2. 每完成 2～3 个关键任务更新一次；
3. 发现高风险、失败、阻塞或范围变化时立即反馈；
4. 任务结束时输出完整看板、耗时汇总和验证结果。

### 1.4 小批量、可验证、可恢复

一次只推进一个原子任务或一个可验证的垂直切片。每个任务必须具备明确输出、验收标准、验证方式和可解释的实际耗时。

## 2. 默认工作流

```text
一句话需求
  ↓
读取 AGENTS.md、Issue / PR 与必要文档
  ↓
建立带耗时列的任务看板
  ↓
DEFINE：确认目标、范围、成功标准和风险
  ↓
PLAN：拆分可独立验证的原子任务
  ↓
BUILD：一次实现一个任务并累计活跃耗时
  ↓
VERIFY：执行测试、构建、契约和静态检查
  ↓
REVIEW：检查正确性、架构、安全、性能和可维护性
  ↓
更新 Draft PR、耗时汇总与最终任务看板
```

默认规则：

- “分析”类任务只分析，不修改代码；
- “实现、修改、修复”类任务使用独立分支和 Draft PR；
- “继续”类任务从现有 PR、CI 和 Review 恢复；
- 能从仓库推断的信息不再重复询问用户；
- 涉及数据库、公共 API、权限、隐私、审核流程或兼容性变化时，必须明确说明影响；
- 不直接修改 `main`。

## 3. 一句话操作指令

### 3.1 分析

```text
分析：来源资料库的搜索、维护、详情和引用关系是否完整。
```

默认行为：读取相关规则、文档和代码，输出当前实现、问题、风险、建议和任务拆解，不修改仓库。

### 3.2 实现

```text
实现：人物录入页面将支派 ID 改为支派名称选择。
```

默认行为：自动定位影响范围、建立看板、创建分支和 Draft PR，按原子任务修改并验证。

也可以直接基于 Issue：

```text
实现 Issue #128。
```

### 3.3 继续

```text
继续 PR #35。
```

默认行为：读取 PR 描述、关联 Issue、diff、提交、CI、Review、耗时和未完成项，先输出恢复检查点，再继续下一个未完成任务。

### 3.4 修复 CI

```text
修复 PR #35 的 CI。
```

默认行为：只聚焦失败检查、失败日志、本批提交和直接相关代码，不重新扫描整个仓库。

### 3.5 检视

```text
检视 PR #35，重点关注权限、API 契约和性能。
```

默认行为：按 Correctness、Readability、Architecture、Security、Performance 五轴输出结论，不修改代码，除非用户明确要求整改。

### 3.6 收尾

```text
收尾 PR #35。
```

默认行为：对照 Issue 验收标准检查任务看板、耗时、diff、测试、CI、文档、风险和遗留项，完善 PR 描述。

### 3.7 查询进度

```text
汇报进度
```

输出完整任务看板和耗时汇总。

```text
只看未完成
```

只展示进行中、阻塞、失败和待处理任务，并保留对应耗时。

## 4. 任务看板规范

### 4.1 状态定义

| 标识 | 状态 | 使用条件 |
|---|---|---|
| ✅ | 已完成 | 已产生结果，并完成必要验证 |
| 🔄 | 进行中 | 当前正在执行或已修改但尚未验证 |
| ⏳ | 待处理 | 尚未开始 |
| ⚠️ | 阻塞 | 因权限、信息、依赖、CI 或技术问题无法继续 |
| ❌ | 失败 | 执行失败且尚未修复 |
| ⏭️ | 已跳过 | 经分析确认无需执行，并写明原因 |

原则上同一时间只保留一个主要任务为“进行中”。

### 4.2 标准看板格式

```markdown
【任务看板】

| 序号 | 任务 | 状态 | 耗时 | 结果或说明 |
|---|---|---|---|---|
| 1 | 读取项目规则与关联需求 | 🔄 进行中 | 已累计约 3 分钟 | 正在确认边界 |
| 2 | 定位相关代码与调用链 | ⏳ 待处理 | — |  |
| 3 | 分析影响范围与风险 | ⏳ 待处理 | — |  |
| 4 | 实现第一个垂直切片 | ⏳ 待处理 | — |  |
| 5 | 补充并执行验证 | ⏳ 待处理 | — |  |
| 6 | 更新 Draft PR | ⏳ 待处理 | — |  |

当前目标：……
下一步：……
```

任务应根据实际需求动态生成，不机械套用固定清单。

### 4.3 耗时口径

“耗时”记录实际活跃执行时间，包括分析、修改、diff 检查、验证操作和结果复核。

以下时间不计入：

- 等待用户决策、权限或补充信息；
- CI 排队和自动运行；
- 外部系统等待；
- 会话中断和明确暂停。

显著外部等待应在“结果或说明”中单独标明。

状态与耗时填写：

- 待处理：`—`；
- 进行中：`已累计约 N 分钟`；
- 已完成：`约 N 分钟` 或 `约 H 小时 M 分钟`；
- 阻塞或失败：保留已累计耗时；
- 规范生效前无法还原：`未记录（规范生效前）`。

不得根据会话时长、Commit 时间差或主观印象补造耗时。

### 4.4 任务拆解标准

不推荐：

- 修改后端；
- 优化前端；
- 完善测试。

推荐：

- 增加来源资料分页查询接口；
- 增加标题和来源类型筛选条件；
- 修改列表页筛选区域；
- 增加详情页引用记录查询；
- 补充搜索接口单元测试；
- 修复前端类型检查失败。

每个任务应满足：

1. 有明确输入与输出；
2. 可以独立判断是否完成；
3. 尽量对应具体文件或提交；
4. 有验收标准与验证方式；
5. 规模足够小，便于 Review、计时和恢复。

### 4.5 完成判定

代码已修改但尚未验证时，仍属于“🔄 进行中”。只有满足以下条件后才可标记为“✅ 已完成”：

- 目标产物已经生成；
- 已检查相关 diff；
- 没有无关修改；
- 必要测试已经补充；
- 能执行的构建、测试或契约检查已经通过；
- 满足 Issue 或用户验收标准；
- 累计活跃耗时已经固化并写回。

### 4.6 动态新增任务

执行中发现新工作时，可以新增任务，但不得静默扩大范围。必须说明：

- 新增原因；
- 是否影响原计划；
- 是否扩大文件或模块范围；
- 是否引入 API、数据库、权限、审核或兼容性风险。

不再需要的任务标记为“⏭️ 已跳过”，不要从看板中删除。未执行的跳过任务耗时填写 `—`。

## 5. 过程反馈规范

### 5.1 反馈模式

| 模式 | 反馈策略 |
|---|---|
| 安静模式 | 仅展示首次看板、阻塞信息和最终看板 |
| 标准模式 | 默认；每完成 2～3 个关键任务更新一次 |
| 密集模式 | 每完成一个关键任务更新一次 |
| 暂停反馈 | 继续执行，但不再发送过程看板 |

用户可直接说：

```text
继续 PR #35，密集模式。
```

未指定时使用标准模式。暂停聊天反馈不等于暂停 GitHub 中的状态和耗时持久化。

### 5.2 进展更新格式

```markdown
【进度更新】

| 序号 | 任务 | 状态 | 耗时 | 结果或说明 |
|---|---|---|---|---|
| 1 | 读取项目规则与需求 | ✅ 已完成 | 约 4 分钟 | 已确认不修改公共 API |
| 2 | 定位列表页、接口和模型 | ✅ 已完成 | 约 7 分钟 | 涉及 6 个文件 |
| 3 | 增加分页搜索接口 | 🔄 进行中 | 已累计约 12 分钟 | 查询逻辑完成，正在补测试 |
| 4 | 修改前端列表页 | ⏳ 待处理 | — |  |

本轮已完成：……
当前正在：……
活跃耗时汇总：……
外部等待：……
发现的问题或风险：……
下一步：……
```

没有外部等待、问题或风险时，可省略对应字段。

### 5.3 必须立即反馈的情况

- CI、构建、测试或契约检查失败；
- 实际实现与需求或文档不一致；
- 缺少权限或必要信息；
- 需要扩大修改范围；
- 需要调整数据库、公共 API、权限、隐私或审核流程；
- 工具、连接或页面异常；
- 需要用户作出关键产品或架构决策。

不得发送虚假进度、虚构剩余时间、补造耗时，或用未验证的百分比表示完成度。

## 6. Draft PR 作为持久化工作区

长任务应尽早建立 Draft PR，而不是等所有代码完成后再创建。

PR 描述至少持续维护：

```markdown
## 目标

## 当前任务看板

| 序号 | 任务 | 状态 | 耗时 | 结果或说明 |
|---|---|---|---|---|

## 已完成

## 当前进行中

## 待处理

## 耗时汇总

## 验证结果

## 已知风险

## 恢复检查点
```

每完成一个原子任务：

1. 检查本批 diff；
2. 删除无关修改；
3. 补充或更新测试；
4. 形成独立提交；
5. 查看 CI；
6. 固化任务耗时；
7. 更新 PR 看板和检查点。

## 7. 页面中断与新会话恢复

恢复任务时，按以下顺序读取：

1. `AGENTS.md`；
2. 关联 Issue；
3. Draft PR 描述；
4. 当前 diff；
5. Commit 记录；
6. CI 状态和失败信息；
7. Review 意见；
8. 上一次任务看板、累计耗时和恢复检查点。

恢复后先输出：

```markdown
【恢复检查点】

| 序号 | 任务 | 状态 | 耗时 | 结果或说明 |
|---|---|---|---|---|
| 1 | …… | ✅ 已完成 | 约 8 分钟 | 不重复执行 |
| 2 | …… | 🔄 进行中 | 已累计约 5 分钟 | 从当前状态继续 |
| 3 | …… | ⏳ 待处理 | — |  |

当前分支：……
当前 PR：……
CI 状态：……
未解决 Review：……
当前任务累计耗时：……
下一步最小任务：……
```

恢复时禁止重复执行已经完成的修改，以 GitHub 实际状态为准。会话中断期间不计入活跃耗时；历史耗时无法确认时填写 `未记录（规范生效前）`。

## 8. 用户最小输入建议

日常使用优先采用以下短指令：

| 场景 | 推荐输入 |
|---|---|
| 只分析 | `分析：<一句业务问题>` |
| 直接实现 | `实现：<一句需求>` |
| 基于需求单 | `实现 Issue #<编号>` |
| 恢复开发 | `继续 PR #<编号>` |
| 修复流水线 | `修复 PR #<编号> 的 CI` |
| 代码评审 | `检视 PR #<编号>，重点关注……` |
| 完成交付 | `收尾 PR #<编号>` |
| 查看全部进度 | `汇报进度` |
| 查看剩余任务 | `只看未完成` |

AI 应主动从仓库中补全默认分支、目录、技术栈、验证命令和项目约束，不要求用户重复填写模板字段。

---

## 9. 适用边界

该工作方式适合：

- 仓库分析；
- 功能开发与页面改造；
- Bug 和 CI 修复；
- PR Review；
- 文档、接口和测试同步；
- 无法使用本地开发环境的远程协作。

以下操作仍必须谨慎处理，并在执行前明确影响：

- 数据库 schema 或迁移变更；
- 公共 API 破坏性调整；
- 权限、隐私、审核流程修改；
- 删除历史兼容逻辑；
- 合并 PR、删除分支和关闭关键 Issue；
- 涉及真实个人隐私或敏感数据的操作。

## 10. 推荐落地组合

```text
AGENTS.md
  定义长期规则
      ↓
Issue
  定义需求和验收
      ↓
任务看板
  展示执行状态与耗时
      ↓
Draft PR
  持久化进度、耗时和风险
      ↓
原子 Commit
  保存每个小批量变更
      ↓
GitHub Actions
  提供可重复验证
      ↓
Review / Merge
  完成人工把关和交付
```

这套模式的核心不是让聊天一次完成所有工作，而是让每次聊天都能从 GitHub 恢复、推进一个明确任务，并把状态、耗时和结果重新写回 GitHub。
