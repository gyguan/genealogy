# 11. MVP1 进展记录 - 2026-07-01

本文记录接续 `docs/10-mvp1-session-handoff.md` 后完成的 MVP1 修复与 P1 增强。

## 1. 已修复：来源附件 Controller 路径冲突

### 问题

后端启动时报错：

```text
Ambiguous mapping:
GET /api/v1/sources/{sourceId}/attachments
SourceEvidenceController#listAttachmentsBySource(Long)
SourceAttachmentController#list(Long)
```

### 处理结果

保留既有证据附件接口：

```http
GET /api/v1/sources/{sourceId}/attachments
```

将新增 `source_attachment` 查询接口迁移到独立命名空间：

```http
GET /api/v1/source-attachments/sources/{sourceId}
```

下载与删除接口保持不变：

```http
GET    /api/v1/source-attachments/{attachmentId}/content
DELETE /api/v1/source-attachments/{attachmentId}
```

同步修改前端 `SourceAttachmentPage` 的附件列表查询路径。

## 2. P1 增强：导入字段自动识别

### 能力说明

人物导入支持 CSV / XLSX 首行表头自动识别，不再完全依赖手工列号。

当前可识别字段别名包括：

| 目标字段 | 支持表头示例 |
| --- | --- |
| 姓名 | 姓名、名字、名讳、name、personName |
| 性别 | 性别、gender、sex |
| 代次 | 代次、世代、第几代、generationNo |
| 字辈 | 字辈、字派、派语、行辈、generationWord |
| 支派ID | 支派ID、分支ID、房支ID、branchId |
| 出生日期 | 出生日期、出生时间、生辰、birthDate、birthday |
| 是否在世 | 是否在世、在世、健在、isLiving、living、alive |

### 接口变化

预览和导入接口新增可选参数：

```http
autoMapping=true|false
```

默认 `true`。当表头识别失败时，继续使用原手工列号作为兜底。

## 4. P1 增强：支派负责人和支派范围权限

### 能力说明

补齐“按支派授权、按支派负责人治理”的可用性闭环：

1. `branch_admin` / `editor` 在 `scopeType=branch` 时，可维护 `scopeId` 指定支派及其下级支派的数据。
2. `clan_admin` 保持宗族级全量维护权限。
3. 成员授权新增/更新时会校验支派范围：`scopeType=branch` 必须指定有效支派，且该支派必须属于当前宗族。
4. 支派维护页支持填写 `managerMemberId`，设置支派负责人。
5. 支派负责人必须是当前宗族有效成员，并且角色必须是 `clan_admin`、`branch_admin` 或 `editor`，同时需要具备该支派范围维护权限。

### 相关代码

- `AuthorizationApplicationService`：支派范围从单支派精确匹配扩展为“当前支派 + 下级支派”。
- `MemberManagementApplicationService`：成员授权时规范化并校验 `scopeType/scopeId/branchId`。
- `MemberApplicationService`：兼容旧成员接口的支派范围校验。
- `BranchApplicationService`：支派负责人设置时校验成员有效性、角色和支派范围权限。
- `BranchPage`：支派管理页面展示并维护负责人成员 ID。
- `MemberPage`：成员授权页面优化“支派范围（含下级支派）”说明与默认值。

## 5. P1 增强：继嗣、出嗣、养父母等复杂关系

### 能力说明

复杂关系从“前端可录入字符串”升级为“前后端都有明确规则”的能力：

1. 后端关系类型白名单增加并规范化：`parent_child`、`spouse`、`adoptive`、`successor`、`out_adoption`。
2. 支持复杂关系标签归一：养父、养母、养父母、继嗣/承嗣/嗣子、出嗣/出继等中文或英文标签会归一为标准标签。
3. 养父、养母关系参与世系循环校验和代次校验，并限制同一子女只能存在一个同类养父或养母关系。
4. 继嗣/承嗣关系作为非血缘世系关系处理，参与世系循环校验和代次校验。
5. 出嗣/出继关系作为事实记录保存，默认不参与本支派世系循环。
6. 更新关系时同样执行复杂关系类型、标签、代次和重复关系校验。
7. 前端关系管理页新增模板化录入，覆盖亲生父母、配偶、养父、养母、养父母/收养、继嗣/承嗣、出嗣/出继。

### 相关代码

- `RelationshipApplicationService`：新增复杂关系类型白名单、标签归一、养父母唯一性、继嗣世系校验、更新场景校验。
- `RelationshipPage`：新增复杂关系模板、录入方向说明、世系/血缘/主关系字段、说明字段和复杂关系展示。

## 6. 当前验证建议

建议下一步本地验证：

```bash
cd backend/genealogy-backend
mvn clean compile
mvn spring-boot:run
```

重点确认：

1. 后端不再出现 `/api/v1/sources/{sourceId}/attachments` ambiguous mapping。
2. `GET /api/v1/source-attachments/sources/{sourceId}` 能正常返回新增附件模型。
3. CSV / XLSX 表头顺序打乱后，预览仍能正确识别姓名、性别、代次等字段。
4. 手动修改导入列号后，前端会关闭自动识别并以手工映射为准。
5. 来源附件图片/PDF/文本类文件可点击“预览”打开。
6. 宗族管理员可新增支派范围成员授权，`scopeId` 必须是当前宗族下的有效支派。
7. 支派范围成员可维护授权支派及其下级支派，不能维护其他支派。
8. 支派负责人只能设置为具备该支派维护权限的有效宗族成员。
9. 养父、养母、继嗣关系能正常预检并创建；代次反向或形成循环时应失败。
10. 出嗣关系能作为非世系事实关系保存，不参与世系循环判断。
