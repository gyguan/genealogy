# 11. MVP1 进展记录 - 2026-07-01

本文记录接续 `docs/10-mvp1-session-handoff.md` 后完成的 MVP1 修复、P1 增强与 P2 产品感增强。

## 1. 已修复：来源附件 Controller 路径冲突

后端启动曾因 `GET /api/v1/sources/{sourceId}/attachments` 在 `SourceEvidenceController` 与 `SourceAttachmentController` 中重复映射而失败。

处理结果：

```http
GET /api/v1/sources/{sourceId}/attachments
GET /api/v1/source-attachments/sources/{sourceId}
GET /api/v1/source-attachments/{attachmentId}/content
DELETE /api/v1/source-attachments/{attachmentId}
```

```text

```

## 2. P1 增强：导入字段自动识别

人物导入支持 CSV / XLSX 首行表头自动识别，不再完全依赖手工列号。预览和导入接口新增可选参数：

```http
autoMapping=true|false
```

默认 `true`。当表头识别失败时，继续使用原手工列号兜底。

## 4. P1 增强：支派负责人和支派范围权限

补齐“按支派授权、按支派负责人治理”的可用性闭环：

1. `branch_admin` / `editor` 在 `scopeType=branch` 时，可维护 `scopeId` 指定支派及其下级支派的数据。
2. `clan_admin` 保持宗族级全量维护权限。
3. 成员授权新增/更新时会校验支派范围。
4. 支派维护页支持填写 `managerMemberId`，设置支派负责人。
5. 支派负责人必须是当前宗族有效成员，且具备该支派范围维护权限。

## 5. P1 增强：继嗣、出嗣、养父母等复杂关系

复杂关系从“前端可录入字符串”升级为“前后端都有明确规则”：

1. 后端关系类型白名单：`parent_child`、`spouse`、`adoptive`、`successor`、`out_adoption`。
2. 支持养父、养母、养父母、继嗣/承嗣/嗣子、出嗣/出继等标签归一。
3. 养父、养母、继嗣关系参与世系循环校验和代次校验。
4. 出嗣/出继关系作为事实记录保存，默认不参与本支派世系循环。
5. 更新关系时同样执行复杂关系类型、标签、代次和重复关系校验。
6. 前端关系管理页新增复杂关系模板化录入。

## 7. P1 增强：人物档案字段补全

人物档案从“最小录入”补齐为“完整档案录入 + 创建状态可控”：

1. `PersonCreateRequest` 新增 `dataStatus` 字段，创建人物时可明确设置 `draft/pending_review/official/rejected/archived`。
2. `PersonMapper` 在创建人物时写入 `dataStatus`，未传时继续由服务层默认补为 `draft`。
3. 建谱向导的人物录入扩展为完整人物档案字段。
4. 新增字段覆盖人物编码、谱名、字号、别名、排行、生卒日期与精度、是否有后裔、地域、履历、墓葬、传记、墓志铭、世系状态、隐私级别、数据状态。

## 9. P2 增强：支派世系图

补齐支派维度的图谱产品能力，不再只能围绕单个人查看世系：

1. 新增后端接口：`GET /api/v1/tree/clans/{clanId}/branches/{branchId}/lineage`。
2. 接口按“当前支派 + 下级支派”收集人物节点。
3. 关系边只保留册内人物之间的世系关系，包含亲子、养育、继嗣等 `isLineageRelation=true` 的关系。
4. 自动计算支派根人物：优先使用支派 `founderPersonId`，否则使用没有父边的最早代次人物。
5. 前端“世系图谱”页面新增“支派世系图”面板，可选择支派后一键生成。
6. 支派世系图按代次分栏展示人物节点，并展示支派人物数、世系关系数、根人物。
7. 点击支派世系图节点可打开人物浮层，也可一键设为中心人物进入原有人物世系树。

## 12. 当前验证建议

建议下一步本地验证：

```bash
cd backend/genealogy-backend
mvn clean compile
mvn spring-boot:run
```

重点确认：

1. 后端不再出现 `/api/v1/sources/{sourceId}/attachments` ambiguous mapping。
2. `GET /api/v1/source-attachments/sources/{sourceId}` 能正常返回新增附件模型。
3. CSV / XLSX 表头顺序打乱后，预览仍能正确识别姓名、性别、代次等字段。
4. 手动修改导入列号后，前端会关闭自动识别并以手工映射为准。
5. 来源附件图片/PDF/文本类文件可点击“预览”打开。
6. 宗族管理员可新增支派范围成员授权，`scopeId` 必须是当前宗族下的有效支派。
7. 支派范围成员可维护授权支派及其下级支派，不能维护其他支派。
8. 支派负责人只能设置为具备该支派维护权限的有效宗族成员。
9. 养父、养母、继嗣关系能正常预检并创建；代次反向或形成循环时应失败。
10. 出嗣关系能作为非世系事实关系保存，不参与世系循环判断。
11. 按支派导出人物时，应包含当前支派及下级支派人物。
12. 按支派导出关系时，只应包含起点和终点人物都在该支派范围内的关系。
13. 建谱向导中填写完整人物档案后创建，详情页应能看到谱名、字号、生卒、地域、履历、墓葬、隐私和数据状态等字段。
14. 创建人物时不传 `dataStatus` 的老链路仍应默认进入 `draft`。
15. 全宗族成册 HTML 下载后应包含封面、支派目录、人物世录和关系索引。
16. 按支派成册 HTML 只应包含该支派及下级支派的人物和内部关系。
17. 浏览器打开成册 HTML 后，可通过打印功能另存为 PDF。
18. 选择支派生成支派世系图时，应只显示该支派及下级支派人物。
19. 支派世系图的边只应连接支派范围内的人物，且按代次分栏展示。
20. 点击支派世系图人物节点，应弹出人物摘要，并可设为中心人物。
21. 族谱首页应展示堂号、郡望、祖籍/发源地、始祖/中心祖、迁徙脉络、支派故事和文化资料摘要。
22. 若支派维护了 `migrationFrom/migrationTo`，首页迁徙脉络应显示真实迁徙路线；否则显示祖籍线索和待维护提示。
23. 追踪中心应能按对象查询操作日志，并生成操作与审核时间线。
24. 输入审核任务 ID 后，应能关联字段级 Diff；若该任务已不在待审列表，也应展示 Diff 和对象日志。
25. 点击日志列表中的“追踪”，应自动带入对象类型和对象 ID。
