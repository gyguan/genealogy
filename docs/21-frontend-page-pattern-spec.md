# Genealogy 前端页面规范

## 1. 目标

针对当前同类型页面在标题、查询区、列表、详情、表单和反馈方式上存在较大差异的问题，建立一套可以直接复用的中后台页面模式。

本规范优先参考 Ant Design 的设计价值观、全局样式、数据录入、数据展示、反馈、按钮、列表页、详情页和表单页设计模式，并结合 Genealogy 当前 React + TypeScript + Ant Design 实现落地。

规范原型入口：

```text
/page-patterns.html
```

本原型为独立入口，不加入正式系统菜单，也不请求真实业务 API。

---

## 2. 设计原则

### 2.1 确定性

- 同一类型页面使用相同结构和相同操作位置。
- 相同状态使用相同颜色、图标、文案和反馈组件。
- 用户在文化资料、迁徙事件和文化场所之间切换时，无需重新学习页面。

### 2.2 信息分层

页面信息按以下层级组织：

1. 页面上下文：标题、说明、宗族范围。
2. 核心任务：新增、提交审核等唯一主操作。
3. 查询和业务内容：筛选、列表、详情、表单。
4. 辅助信息：规则提示、追踪、帮助说明。
5. 异常与反馈：就近展示，不覆盖主要内容。

### 2.3 直接与克制

- 一个页面只保留一个视觉主操作。
- 常用筛选条件直接展示，低频条件放入“更多筛选”。
- 表格行内只展示高频动作，其余进入“更多”。
- 不向业务用户暴露数据库 ID、接口字段名、技术错误堆栈。

### 2.4 即时反馈

- 表单错误显示在字段附近。
- 查询错误显示在列表区域内。
- 操作成功使用轻量 `message.success`。
- 操作失败保持弹窗或表单打开，不丢失用户输入。
- 禁止通过未处理 Promise 或抛异常代替表单校验。

## 3. 全局设计基线

| 项目 | 桌面端 | 移动端 | 说明 |
|---|---:|---:|---|
| 页面内容外边距 | 24 px | 12 px | 应用外壳统一控制 |
| 页面模块间距 | 16 px | 12 px | 页面头、查询卡片、列表卡片之间 |
| Card 内容边距 | 16 px | 16 px | 页面头可使用 20～24 px |
| 表单项间距 | 12 px | 12 px | 通过主题统一配置 |
| 标准控件高度 | 32 px | 40 px | 桌面默认使用 Ant Design `middle`，移动端保证触控友好 |
| 页面内容最大宽度 | 1600 px | 100% | 宽屏居中，避免无限拉伸 |
| 标准圆角 | 8 px | 8 px | 大 Card 可使用 12 px |
| 详情抽屉宽度 | 720 px | 100vw | 复杂对象可使用 760～800 px |
| 查询列表表格 | middle | small/middle | 移动端优先改为卡片列表 |

### 3.1 页面主操作

- 页面主操作位于页面头右上角。
- 每个页面只允许一个 `type="primary"` 的主操作。
- 宗族选择器属于上下文，不属于主操作。
- 次要操作使用默认按钮或链接按钮。
- 删除等危险操作不得使用主色按钮。

### 3.2 标题层级

| 层级 | 用途 | 推荐实现 |
|---|---|---|
| 页面标题 | 页面任务名称 | `Typography.Title level={3}` |
| Card 标题 | 查询区、列表区、信息分组 | `Card title` |
| 详情标题 | 当前对象名称 | `Typography.Title level={4}` |
| 内容小节 | 摘要、正文、附件等 | `Typography.Title level={5}` 或小 Card |

### 3.3 按钮规范

#### 3.3.1 尺寸与点击热区

优先使用 Ant Design 内置的 `small`、`middle`、`large` 三档尺寸，不在业务页面中自行定义零散高度。

| 按钮类型 | 推荐尺寸 | 最小宽度 | 使用场景 |
|---|---:|---:|---|
| 页面主按钮 | 32 px；重点任务可 40 px | 88 px | 新增、创建、提交审核、确认发布 |
| 普通操作按钮 | 32 px | 72 px | 查询、重置、保存草稿、取消、返回 |
| 表格文字按钮 | 32 px 高的操作热区 | 不固定 | 查看、编辑等行内高频动作 |
| 纯图标按钮 | 32 × 32 px | 32 px | 更多、刷新、关闭、展开 |
| 移动端按钮 | 40 px | 88 px 或等宽 | 页面头和底部固定操作区 |

- 触屏场景的实际可点击热区不得小于 `44 × 44 px`，可以通过外层容器或内边距实现。
- 同一操作组中的按钮高度必须一致。
- 不允许通过业务 CSS 将普通按钮压缩到 28 px、30 px 等非标准尺寸。
- 文案较长时优先调整布局或移入“更多”，不得通过缩小字号和高度强行容纳。

#### 3.3.2 语义层级

| 层级 | Ant Design 实现 | 使用规则 |
|---|---|---|
| 主操作 | `type="primary"` | 一个页面或一个弹窗最多一个 |
| 次要操作 | 默认按钮 | 保存草稿、重置、返回、取消等 |
| 轻量操作 | `type="link"` 或 `type="text"` | 表格查看、编辑、帮助、展开等 |
| 危险操作 | `danger` | 删除、清除、不可逆操作 |
| 不可用操作 | `disabled` | 仅用于用户可理解且未来可能恢复的状态 |

- 不能同时出现两个视觉权重相同的主色按钮。
- “保存”和“提交审核”同时存在时，“提交审核”为主操作，“保存草稿”为默认按钮。
- 危险操作不能使用主色伪装成普通确认，也不能与主操作紧邻形成误触风险。
- 无权限且用户无法申请或恢复权限的操作直接隐藏；因状态暂时不可执行且用户需要知道其存在时才禁用，并提供原因说明。

#### 3.3.3 摆放位置与操作顺序

按钮遵循“次要在左、主要在右”的统一顺序。

| 页面区域 | 统一位置 | 推荐顺序 |
|---|---|---|
| 页面头 | 右上角 | 上下文选择器、次要操作、主操作 |
| 查询 Card | 右下角 | 重置、查询 |
| 列表工具栏 | Card 标题右侧 | 批量操作、视图设置、刷新 |
| 表格操作列 | 固定在最右侧 | 查看、编辑、更多 |
| 详情抽屉 Header | 标题右侧 | 追踪、编辑、提交审核、更多 |
| 编辑页底部固定栏 | 右侧 | 取消、保存草稿、提交审核 |
| Modal Footer | 右侧 | 取消、确认 |
| 空态/错误态 | 状态说明下方居中 | 次要操作、主操作 |

- 返回操作优先使用页面头左侧的返回入口或面包屑，不与右侧主操作混排。
- 同一页面不得在 Header、正文末尾和底部同时重复完全相同的一组按钮。
- 列表页“新增”只放在页面头，除首次空态引导外，不在查询 Card 和列表 Card 重复放置。
- Modal 确认按钮始终位于最右侧；危险确认使用 `danger`，取消始终位于其左侧。

#### 3.3.4 间距与分组

- 同一按钮组的标准间距为 `8 px`，推荐使用 `<Space size={8}>` 或等价 Token。
- 纯图标按钮之间可使用 `4 px`，但与文字按钮之间保持 `8 px`。
- 不同语义组之间使用 `16 px` 间隔，或通过分隔线、“更多”菜单形成分组。
- 危险操作若必须直接展示，应与普通操作至少间隔 `16 px`；默认优先收进“更多”。
- 页面头按钮组与标题说明之间至少保留 `24 px`，空间不足时允许换行，禁止挤压标题。
- 表格行内操作应保持单行，不因按钮换行导致行高不一致。

#### 3.3.5 图标使用

- 图标用于提升识别效率，不用于装饰；同类操作必须保持一致。
- 页面级“新增”“查询”“刷新”等高频动作可以使用前置图标。
- “取消”“确定”“保存草稿”等语义已清晰的按钮默认不强制添加图标。
- 表格中的“查看、编辑”优先使用文字按钮，避免整列只剩难以识别的图标。
- 纯图标按钮必须提供 `aria-label` 和 Tooltip，不能依赖用户猜测图标含义。
- 同一按钮中只使用一个主图标，图标固定在文字左侧；“下一步”等方向性操作可放在右侧。

#### 3.3.6 Loading、Disabled 与重复提交

- 发起异步操作的按钮必须使用 `loading` 状态，而不是仅修改按钮文案。
- Loading 时保留按钮原有宽度，避免布局跳动。
- 提交过程中禁用取消、关闭和其他可能改变表单数据的操作；是否允许取消请求由具体业务决定。
- 同一操作不得被重复触发；服务端仍需保证幂等或进行状态校验。
- Disabled 按钮需要说明原因时，使用 Tooltip 或邻近说明，不在点击后才提示“不可用”。
- 请求失败后恢复按钮可用状态，并保留用户输入和当前上下文。

#### 3.3.7 文案与宽度

- 按钮使用动宾短语，例如“新增资料”“提交审核”“重新加载”，避免只写“确定”“处理”。
- 同一业务动作在不同页面保持相同文案，不混用“保存”“确认保存”“提交保存”。
- 普通按钮文案建议控制在 2～6 个汉字；超过 8 个汉字时优先调整文案或放入菜单。
- 按钮文字默认不换行；按钮组空间不足时整体换行或折叠低频操作。
- 不为追求等宽而设置过大的固定宽度；仅移动端底部操作栏或成组选择场景允许等宽。

#### 3.3.8 移动端规则

- 页面主按钮和底部确认按钮高度使用 40 px，实际触控热区不小于 44 px。
- 页面头操作超过两个时，保留唯一主操作，其余进入“更多”。
- 底部固定栏可以采用等宽按钮；存在三个操作时，主按钮可以占更大比例。
- 表格或卡片行内最多直接展示“查看”和“更多”，编辑、审核、归档等进入菜单。
- 危险操作不得与主操作紧贴，必要时进入独立确认页或二次确认弹窗。

#### 3.3.9 推荐实现

```tsx
<Space size={8}>
  <Button>取消</Button>
  <Button>保存草稿</Button>
  <Button type="primary" loading={submitting}>
    提交审核
  </Button>
</Space>
```

```tsx
<Space size={4}>
  <Button type="link">查看</Button>
  <Button type="link">编辑</Button>
  <Tooltip title="更多操作">
    <Button type="text" icon={<MoreOutlined />} aria-label="更多操作" />
  </Tooltip>
</Space>
```

### 3.4 页面内容分区与 Tabs 规范

Tabs 用于在同一页面上下文中组织多个并列、相对独立且需要频繁切换的内容区域。不能仅因为页面较长，就机械地将每个 Card 转成一个 Tab。

#### 3.4.1 组件选择决策

| 内容特征 | 推荐模式 | 说明 |
|---|---|---|
| 1～2 个较短信息分组 | Card 分组或锚点 | 保持信息连续，避免无意义切换 |
| 3～5 个并列业务分区 | Tabs | 适合基本信息、来源附件、审核追踪等同一对象分区 |
| 6～7 个稳定业务分区 | Tabs + “更多”或重新归类 | 优先合并低频分区，避免标签栏拥挤 |
| 超过 7 个分区 | 二级导航或独立子页面 | 说明信息架构已超出单页 Tabs 承载范围 |
| 存在明确先后顺序 | Steps / 分步表单 | Tabs 不表达流程先后和完成状态 |
| 属于不同对象或不同核心任务 | 路由、菜单或独立页面 | 不应伪装成同一页面的并列内容 |
| 仅用于展开/收起补充内容 | Collapse | 不使用 Tabs 隐藏少量辅助信息 |

满足以下条件时优先使用 Tabs：

- 所有分区共享同一个页面标题、业务对象和权限上下文；
- 分区之间是并列关系，而不是严格的先后步骤；
- 用户需要在分区之间频繁切换和对照；
- 每个分区可以独立加载、展示空态和处理错误；
- 单页连续滚动会明显降低定位效率或造成首屏信息过载。

以下情况不使用 Tabs：

- 只有一个有效分区，或第二个分区只是几行补充信息；
- 通过 Tabs 掩盖字段分组混乱、组件重复或信息架构不清；
- 用户必须按顺序完成任务；
- 不同 Tab 实际对应不同权限域、不同对象或不同页面主任务；
- 需要嵌套两层及以上 Tabs。原则上禁止嵌套 Tabs，确有必要时应重新拆分页面。

#### 3.4.2 标准页面结构

```text
页面头
  ├─ 面包屑、标题和说明
  ├─ 当前对象/宗族上下文
  └─ 页面级主操作
对象摘要或全局提示（可选）
Tabs 导航
  ├─ 基本信息
  ├─ 业务数据
  ├─ 来源与附件
  └─ 审核与追踪
当前 Tab 内容区
  ├─ Tab 级工具栏
  ├─ 数据内容
  └─ 当前 Tab 的加载、空态或错误反馈
```

- 页面标题、对象状态、全局摘要和页面级主操作放在 Tabs 之外，切换 Tab 时保持稳定。
- Tabs 位于页面头或对象摘要之后、具体内容之前，不放在页面最底部或 Card 深层内部。
- 页面级提示只展示一次；只影响某个分区的提示放在对应 Tab 内。
- 页面级操作不在每个 Tab 重复；Tab 专属操作放在当前 Tab 内容区右上角。
- 详情抽屉使用 Tabs 时，Tabs 紧跟 Drawer Header；不再额外套一层无意义的大 Card。

#### 3.4.3 Tab 数量、命名与顺序

- 推荐展示 `2～5` 个 Tab，桌面端直接展示不得超过 `7` 个。
- Tab 名称使用稳定的业务名词，建议控制在 `2～6` 个汉字，例如“基本信息”“来源附件”“审核追踪”。
- 不使用“点击查看”“进行编辑”等动作型长文案作为 Tab 名称。
- 顺序保持稳定：概览或基本信息在最左侧，高频业务内容居中，附件、日志、追踪等辅助内容靠后。
- 相同对象类型在不同页面中使用相同 Tab 名称和顺序，禁止随页面任意调整。
- 数量徽标只用于有明确业务价值的未读数、待处理数或对象数；不为每个 Tab 默认展示 `0`。
- 状态提示优先使用 Badge 或简短数字，不能把完整错误文案塞进 Tab 标签。

#### 3.4.4 URL、切换与状态保持

- 当前激活 Tab 必须可通过 URL 恢复，推荐使用查询参数 `tab=<key>`。
- 浏览器前进、后退和刷新后应保持当前 Tab，不得总是跳回第一个 Tab。
- 可分享链接应直接打开目标 Tab；无效或无权限的 Tab Key 回退到第一个可访问 Tab。
- 切换 Tab 不应重置页面级筛选、对象上下文和已加载的全局摘要。
- 列表 Tab 应分别保留各自的分页、排序和筛选状态，返回时避免用户重新操作。
- 存在未保存编辑内容时，切换 Tab 前必须保留表单状态；若切换会导致内容销毁，应先提示用户确认。
- 不允许通过 Tab 切换绕过未保存变更提示、校验或权限判断。

#### 3.4.5 加载、空态与错误隔离

- 默认只加载激活 Tab，避免进入页面时同时请求全部低频数据。
- 已加载 Tab 再次切换回来时，应根据数据时效选择缓存或重新请求，禁止无条件重复加载。
- 每个 Tab 独立展示 `Skeleton`、`Spin`、空态、403、404 和服务失败状态。
- 单个 Tab 加载失败不得阻断页面头、对象摘要和其他 Tab。
- 切换过程中需要取消旧请求或忽略过期响应，避免慢请求覆盖当前 Tab 数据。
- Tab 标签可以展示待处理数量，但不得将后台请求错误伪装成数量为零。

Ant Design 实现建议：

- 纯展示型低频内容可采用懒加载，并按需要使用 `destroyOnHidden`。
- 表单、草稿或需要保持局部交互状态的 Tab 默认不销毁，或将状态提升至页面级 Store/Form。
- 是否销毁必须根据状态保持需求决定，不得全局一刀切。

#### 3.4.6 操作与主按钮规则

- 页面级主操作仍然只保留一个，放在页面头右上角，不随 Tab 重复出现。
- Tab 专属操作放在当前内容区或 Card Header 右侧，切换后同步变化。
- 当前 Tab 内也只能有一个视觉主操作；低频操作进入“更多”。
- 涉及多个 Tab 数据的“保存全部”“提交审核”放在页面底部固定栏或页面头，不放在某个 Tab 内造成作用域误解。
- 删除、归档等危险操作仍遵循危险操作规范，不因为空间不足直接堆叠在 Tab 标签栏右侧。
- Tab 标签本身只负责导航，不在标签中放置新增、删除、刷新等按钮。

#### 3.4.7 复杂表单的 Tabs 使用

复杂编辑页可在字段较多且分组相对独立时使用 Tabs，但必须同时满足：

- 各分组属于同一个提交对象；
- 分组之间不存在严格的填写顺序；
- 用户可以理解各分组名称和归属；
- 页面仍提供统一保存或提交入口。

表单 Tab 规则：

- 提交时校验全部 Tab，而不是只校验当前 Tab。
- 校验失败后自动切换到第一个存在错误的 Tab，并滚动到首个错误字段。
- 非当前 Tab 存在错误时，在 Tab 标签上展示错误标识或错误数量。
- 必填核心字段优先放在第一个“基本信息”Tab，不将所有关键输入隐藏在低频 Tab。
- 切换 Tab 不清空字段值，不重复初始化表单默认值。
- 有未保存修改时离开页面仍需统一确认，不能按 Tab 分散提示。
- 存在明确先后依赖的多步骤录入使用 `Steps`，不使用 Tabs 模拟向导。

#### 3.4.8 查询列表页和详情页的 Tabs

查询列表页使用 Tabs 时：

- 仅用于同一对象集合的稳定业务视图，例如“全部、待审核、已归档”；临时条件仍使用筛选器或 `Segmented`。
- 不为每一种状态都复制一套查询表单；公共筛选区放在 Tabs 之外，Tab 只附加稳定范围条件。
- 各 Tab 的数量必须与当前权限和筛选范围一致，避免显示不可访问对象数量。
- Tab 切换应恢复对应分页和排序，不能把上一个 Tab 的页码错误带入新视图。

详情页或详情抽屉使用 Tabs 时：

- 推荐顺序为“基本信息 → 业务明细 → 来源附件 → 审核追踪”。
- 对象名称、状态、更新时间和高频操作放在 Header，不随 Tab 切换。
- Tracking、附件等辅助 Tab 失败不阻断基本信息。
- 内容过长、子表过多或需要长期工作入口时，升级为独立详情页，不继续增加 Tab。

#### 3.4.9 移动端规则

- 移动端 Tab 标签保持单行，使用横向滚动，不允许标签自动换成两行。
- 首屏建议直接可见 `3～4` 个 Tab，低频项可进入“更多”或改用下拉选择器。
- 禁止在移动端嵌套 Tabs；复杂层级改为独立子页面。
- Tab 点击热区高度不小于 `44 px`，激活状态必须同时通过文字/指示条体现，不能只依赖颜色。
- 标签栏可以在长内容页面中吸顶，但不得覆盖页面头、系统导航或表单错误提示。
- 移动端切换 Tab 时保持原滚动位置还是回到内容顶部，应按页面类型统一：详情型默认回到内容顶部，列表型恢复各自滚动位置。

#### 3.4.10 推荐实现

```tsx
const tabItems = [
  { key: 'overview', label: '基本信息', children: <OverviewTab /> },
  { key: 'sources', label: '来源附件', children: <SourceTab /> },
  { key: 'tracking', label: '审核追踪', children: <TrackingTab /> }
];

<Tabs
  activeKey={activeTab}
  items={tabItems}
  onChange={nextTab => updateUrl({ tab: nextTab })}
  destroyOnHidden={false}
/>
```

表单错误跨 Tab 定位示例：

```tsx
try {
  await form.validateFields();
  await submit();
} catch (errorInfo) {
  const firstField = errorInfo.errorFields?.[0]?.name?.[0];
  const targetTab = fieldTabMap[firstField];
  if (targetTab) setActiveTab(targetTab);
}
```

## 4. 查询列表页规范

### 4.1 标准结构

```text
页面头
  ├─ 面包屑
  ├─ 页面标题和说明
  ├─ 当前宗族/范围
  └─ 唯一主操作
查询 Card
  ├─ 常用筛选条件
  ├─ 更多筛选入口
  └─ 重置 / 查询
结果 Card
  ├─ 标题和总数
  ├─ 批量操作或视图设置
  ├─ 状态反馈
  ├─ Table / Card List
  └─ 分页
```

### 4.2 查询区

- 使用 `Form layout="vertical"`。
- 桌面端每行建议 4～6 个筛选项。
- 常用条件不超过 8 个。
- 查询按钮位于查询区右下角。
- 操作顺序固定为“重置、查询”。
- 查询按钮使用主色，重置使用默认按钮。
- 查询成功不弹成功消息，列表刷新本身就是反馈。
- 查询失败在结果 Card 内展示 `Alert` 或 `Result`。

### 4.3 列表区

- Card 标题使用“对象名称 + 总数”。
- 表格名称列放在最左侧，并承担进入详情的主入口。
- 操作列固定在右侧。
- 默认行内动作不超过 3 个：查看、编辑、更多。
- 审核、归档、删除等低频动作进入“更多”，或在详情中执行。
- 不建议整行作为唯一点击入口；若支持整行点击，仍需保留可聚焦的名称链接。

### 4.4 状态

#### 正常态

展示当前筛选范围和业务规则提示，避免使用干扰性强的持续警告。

#### 空态

必须说明：

- 为什么没有数据；
- 用户下一步可以做什么；
- 是否可以重置筛选或新增对象。

#### 首次加载失败

- 使用 `Result status="error"` 或错误 `Alert`。
- 提供“重新加载”。
- 不展示虚假的空数据。

#### 刷新失败

- 保留上次成功结果。
- 使用 warning `Alert` 说明当前展示的是上次数据。

#### 无权限

- 使用 `Result status="403"`。
- 不展示受限对象名称、数量、摘要或其他可推断身份的信息。

## 5. 详情规范

### 5.1 详情抽屉适用条件

满足以下条件时使用抽屉：

- 用户需要保留列表上下文；
- 详情以查看为主；
- 字段数量不超过约 18 个；
- 子任务较少；
- 页面切换成本高于抽屉承载成本。

以下情况升级为独立详情页：

- 存在复杂编辑、审核或协作任务；
- 多个大型子表；
- 需要 URL 作为长期工作入口；
- 内容过长或需要大面积可视化。

### 5.2 抽屉结构

```text
Header
  ├─ 对象名称
  ├─ 当前状态
  ├─ 编号和更新时间
  └─ 高频操作（最多 3 个）
Tabs
  ├─ 基本信息
  ├─ 来源与附件
  └─ 审核与追踪
Content
  ├─ Descriptions
  ├─ 摘要和正文
  ├─ 子对象列表
  └─ 空态/错误态
```

### 5.3 操作位置

- 查看追踪、编辑和提交审核位于抽屉 Header。
- 归档、删除等低频或危险操作放入“更多”。
- 移动端动作可以折叠或改为底部操作区。
- 不在详情正文末尾重复一套完全相同的操作按钮。

### 5.4 错误状态

必须区分：

- 加载中：`Skeleton`；
- 404：对象不存在或已删除；
- 403：无权查看；
- 网络/服务失败：错误 `Alert` 和重试；
- Tracking 失败：只影响追踪分组，不阻断基本详情。

## 6. 编辑表单规范

### 6.1 Modal 与独立页面选择

| 条件 | Modal | 独立编辑页 |
|---|---:|---:|
| 字段数量 ≤ 12 | 推荐 | 可选 |
| 字段数量 > 12 | 不推荐 | 推荐 |
| 需要分组说明 | 不推荐 | 推荐 |
| 存在长正文 | 不推荐 | 推荐 |
| 多步骤任务 | 不适用 | 推荐向导或 Steps |
| 快速修改少量属性 | 推荐 | 不推荐 |

### 6.2 独立编辑页结构

```text
页面头：返回、标题、说明
状态 Alert：正式对象变更规则、驳回原因等
基本信息 Card
内容信息 Card
治理与展示 Card
底部固定操作区：取消、保存草稿、提交审核
```

### 6.3 表单布局

- 桌面端最多 3 列。
- 长文本始终独占整行。
- 业务上相关字段放在同一分组。
- 不同安全等级、可见范围和可信度放在“治理与展示”。
- 人物关联必须使用业务搜索选择器，禁止要求用户输入人物数据库 ID。
- 经纬度等成对字段使用联合校验。

### 6.4 校验与提交

- 使用 `Form.Item rules` 就地校验。
- Modal 使用 `form.submit()` 和 `onFinish`。
- `validateFields()` 必须处于可捕获的调用链中。
- 校验失败不显示全局错误消息。
- 请求失败后表单保持打开并保留输入。
- 保存期间禁用关闭、重复提交和可变操作。

### 6.5 底部操作区

操作顺序固定：

```text
取消 | 保存草稿 | 提交审核
```

- “提交审核”是主操作。
- “保存草稿”是默认按钮。
- 取消不会触发保存。
- 页面有未保存修改时，离开需要确认。

## 7. 审核、归档与删除规范

### 7.1 统一结构

```text
弹窗标题
影响对象和当前状态
操作后果 Alert
操作原因或提交说明
取消 / 确认
```

### 7.2 操作分级

| 操作 | 语义色 | 原因 | 确认文案 |
|---|---|---|---|
| 提交审核 | 主色/信息色 | 选填 | 确认提交 |
| 归档草稿 | 默认/警告 | 必填 | 确认归档 |
| 申请归档正式对象 | 默认/警告 | 必填 | 提交归档申请 |
| 删除草稿 | 危险色 | 可按规则必填 | 确认删除 |
| 申请删除正式对象 | 危险色 | 必填 | 提交删除申请 |

### 7.3 反馈

- 成功：关闭弹窗、显示成功消息、刷新对象。
- 失败：弹窗保持打开、保留原因、展示服务端错误。
- 状态冲突：说明对象已被他人修改，提供刷新入口。
- 权限失败：不继续重试，并隐藏后续不可用动作。

## 8. 响应式规范

### 8.1 断点策略

| 宽度 | 页面策略 |
|---|---|
| ≥ 1200 | 完整查询网格、Table、720 px Drawer |
| 768～1199 | 查询项减少列数、表格横向滚动 |
| < 768 | 12 px 页面边距、单列筛选、详情占满视口 |

### 8.2 移动端列表

横向表格只能作为过渡方案。核心业务页面应逐步支持 Card List：

```text
对象名称
支派 / 类型
状态 + 可信度
更新时间
查看 / 更多
```

### 8.3 移动端操作

- 页面头按钮允许换行并占满宽度。
- 抽屉 Header 动作折叠为两行或“更多”。
- 长表单保持单列。
- 底部操作区按钮等宽或保留唯一主按钮。
- 可见按钮高度使用 40 px，实际触控热区不小于 44 px。
- 页面头直接展示的操作不超过两个，其余收进“更多”。

## 10. 页面验收清单

### 通用按钮

- [ ] 页面或弹窗只有一个主色按钮。
- [ ] 桌面端按钮使用 32 px 标准高度，移动端可见按钮使用 40 px，触控热区不小于 44 px。
- [ ] 同一按钮组高度一致，按钮间距默认 8 px。
- [ ] 操作顺序符合“次要在左、主要在右”。
- [ ] 表格行内直接操作不超过“查看、编辑、更多”三个。
- [ ] 纯图标按钮具备 Tooltip 和 `aria-label`。
- [ ] 异步提交具备 Loading 状态并阻止重复提交。
- [ ] 无权限操作按规则隐藏，状态性不可用操作才禁用并说明原因。
- [ ] 危险操作使用 `danger`，并与普通操作进行分组或放入“更多”。
- [ ] 按钮文案使用明确动宾短语，同一动作全局文案一致。

### 页面分区与 Tabs

- [ ] Tabs 只用于同一对象、同一页面任务下的并列内容，不用于表达流程步骤。
- [ ] 使用 Tabs 前已比较 Card 分组、锚点、Collapse、Steps 和独立子页面等方案。
- [ ] 推荐 2～5 个 Tab，直接展示不超过 7 个，且不存在嵌套 Tabs。
- [ ] Tab 命名为简短稳定的业务名词，同类页面顺序一致。
- [ ] 页面标题、全局摘要和主操作位于 Tabs 之外，未在每个 Tab 重复。
- [ ] 当前 Tab 可通过 URL 恢复，浏览器前进、后退和刷新行为正确。
- [ ] 各 Tab 独立处理加载、空态、错误和权限，单个 Tab 失败不阻断其他内容。
- [ ] 切换 Tab 能保持必要的分页、筛选、表单和草稿状态。
- [ ] 表单提交会校验全部 Tab，并能定位到第一个错误 Tab。
- [ ] 移动端标签保持单行可滚动，点击热区不小于 44 px，低频 Tab 有收纳策略。

### 查询列表页

- [ ] 页面头位置和层级符合统一模板。
- [ ] 页面只有一个主操作。
- [ ] 查询与重置位置统一，顺序为“重置、查询”。
- [ ] 正常、空态、错误、刷新失败和无权限均有实现。
- [ ] 列表操作不超过 3 个直接动作。
- [ ] URL 可恢复筛选、分页和选中对象。

### 详情

- [ ] 加载、404、403、服务失败状态区分明确。
- [ ] 高频操作位于 Header，直接展示不超过 3 个。
- [ ] 信息按业务分组而不是按接口结构排列。
- [ ] Tracking 失败不阻断基本详情。
- [ ] 移动端抽屉占满视口，低频动作进入“更多”。

### 编辑

- [ ] Modal/独立页选择符合字段复杂度。
- [ ] 字段按业务语义分组。
- [ ] 所有必填规则有中文提示。
- [ ] 校验失败不会产生全局未处理异常。
- [ ] 提交失败保留输入。
- [ ] 存在未保存变更离开提示。
- [ ] 底部操作顺序为“取消、保存草稿、提交审核”。

### 治理操作

- [ ] 显示影响对象和操作后果。
- [ ] 原因必填使用 Form 校验。
- [ ] 删除操作使用危险色。
- [ ] 弹窗操作顺序为“取消、确认”，确认按钮位于最右侧。
- [ ] 成功和失败反馈符合统一规则。
- [ ] 正式对象操作明确说明需要审核。

## 11. 原型评审建议

评审时重点确认以下问题：

1. 查询页的页面头、查询区和列表区是否适合全部中后台模块。
2. 详情默认使用 720 px 抽屉是否满足文化资料、迁徙和场所。
3. 长表单是否由现有 Modal 调整为独立编辑页。
4. 行内动作是否统一收敛为“查看、编辑、更多”。
5. 移动端是否接受逐步从 Table 转向 Card List。
6. 审核、归档和删除是否统一使用一套治理弹窗。
7. 按钮尺寸、间距、顺序、Loading 和移动端热区是否可以作为全局强制基线。
8. Tabs 的适用边界、数量、URL 状态和移动端收纳策略是否可以作为全局基线。

确认本规范后，再将共享模板组件应用到宗族文化三个 Tab，并逐步推广到人物档案、来源资料库、成员与权限等模块。
